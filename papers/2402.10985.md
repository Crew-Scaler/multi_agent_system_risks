## ABSTRACT

Cloud computing services provide scalable and cost-effective solutions for data storage, processing, and collaboration. With their growing popularity, concerns about security vulnerabilities are increasing. To address this, fi rst , we provide a formal model, called CloudLens, that expresses relations between different cloud objects such as users, datastores, security roles, representing access control policies in cloud systems. Second , as access control misconfigurations are often the primary driver for cloud attacks, we develop a planning model for detecting security vulnerabilities. Such vulnerabilities can lead to widespread attacks such as ransomware, sensitive data exfiltration among others. A planner generates attacks to identify such vulnerabilities in the cloud. Finally , we test our approach on 14 real Amazon AWS cloud configurations of different commercial organizations. Our system can identify a broad range of security vulnerabilities, which state-of-the-art industry tools cannot detect.

Cloud IAM configurations (e.g., for Amazon AWS, Google GCP) are often defined using a collection of several JSON files that represent IAM policies. IAM policies encode the set of permissions that decide which user or cloud identity can access what resources (e.g., datastores). Such IAM policies create a web of identities and security permissions, which is impractical for IT admins to analyze manually. Furthermore, over time, as a result of organization changes, IAM policies move away from the principle of least privileges, and may unintentionally allow non-admin users access to sensitive data. Therefore, our goal is (a) to formally model the network of cloud identities, resources and security permissions (called CloudLens ) that allows reasoning about various security tasks (e.g., which users have access to what sensitive data, what types of attacks can happen such as ransomware, sensitive data exfiltration); (b) develop a tool called CloudExploit , that can generate a range of security attacks in a given IAM configuration to allow vulnerability analysis and network hardening. These contributions are critical for penetration testing to harden IAM policies to common attacks such as privilege escalation, sensitive data exfiltration among others.

Related work . There are several types of known attacks that exploit the misconfiguration of clouds such as privilege escalation attacks (a non-admin entity getting admin permissions), impact attacks (deleting data buckets), ransomware attacks (encrypting sensitive data), data exfiltration (stealing private sensitive data) among others [10]. There has been some analysis of what kind of common IAM misconfigurations can cause such attacks [13]. There are relatively few open source tools 1 that analyze a cloud configuration to detect vulnerabilities such as PMapper [28]. However, such tools do not perform a systematic search to detect complex vulnerabilities, and empirically are unable to detect a wide range of attacks our system can detect. Tools such as Policy Sentry [36] are preventative tools that help ease the complexity of writing secure IAM Policies. The problem of optimizing IAM policies using constraint programming has been addressed in [21]. However, their method computes a new IAM policy rather than analyzes an existing one.

Formal methods have been used to analyze the security aspect of cloud offerings by cloud providers such as Amazon. The Amazon offering Zelkova [6], an IAM policy analysis tool, can reason whether a data bucket has public access using SMT solver. Microsoft uses theorem provers to validate global properties of their data centers [19]. A key limitation of these tools is that they analyze the current state of the organization. In contrast, CloudExploit can reason about multi-step attacks that involve an attacker performing a sequence of operations to compromise the system. Another model checking method is presented in [18]. However, this tool is specifically focused on privilege escalation (PE) attacks and can only provide a binary answer whether the current IAM configuration allows PE attacks or not.

Other modeling efforts include [34] and [38]. The former models only the NTFS file system, which is similar to modeling AWS S3 object storage within CloudExploit, however, CloudExploit goes beyond modeling S3 service. It accurately models permission flows that allow it to find attack paths that stem from an attacker's ability to manipulate IAM policies (e.g. during privilege escalation or lateral movement). On the other hand, [38] is concerned with userrole reachability, which solves the problem of finding a path that reaches some admin role in administrative role-based access control policies (ARBAC), distinct from the role-based access control (RBAC) policies analyzed by CloudExploit. Neither of these models addresses permission flows that allow attackers to assume multiple roles, obtain group memberships, or gain elevated permissions.

Attacker emulation tools such as Caldera [27] primarily focus on automated adversary emulation within an enterprise network context. Its primary goal is to simulate the actions of an actual attacker, making it useful for testing network defenses . Caldera is not specifically designed to assess individual components, such as AWS IAM policies.

In the AI literature, simulated penetration testing (pentesting), for example of networked systems, is closely related [9, 16, 30, 37]. In such a setting, a model of the underlying system is constructed and using model-based planning [15] attacks are generated [16]. Attack graphs have also been used to model complex security scenarios using a graphical network of exploits and security conditions [24, 29, 41]. We will show how CloudLens can be used to construct such attack graphs for the IAM security model.

## Contributions . Our key contributions are:

- We develop a simple and generic representation of a cloud IAM, called CloudLens , which represents security permissions, cloud identities and resources using a relation tuplebased formulation, which has been shown to be highly efficient and scalable for authorization control in large-scale clouds [32]. Key benefits of this formulation include (a) It is easily extendable to allow future cloud configuration changes; (b) it is explainable as IT admins can directly change the text file storing such tuples; (c) it is general enough to represent multiple cloud platforms such as Amazon AWS, Google GCP, Microsoft Azure among others.
- We show how the problem of generating attacks (such as sensitive data exfiltration, ransomware etc) can be translated into a planning formulation [26] with appropriate actions. Our approach is able to detect multi-step attacks. We also provide theoretical analysis of the IAM attack problem and show it is NP-Hard.
- Finally, we test on 14 real-world cloud instances and show that our approach can detect a range of attacks. In contrast, industry tools such as PMapper are unable to generate such a diverse range of attacks [28].

## 2 CLOUD IDENTITY ACCESS MANAGEMENT

For ease of exposition, we focus on Amazon AWS, and briefly describe relevant concepts; more details can be found here [3]. AWS supports both role-based access control (RBAC) and attribute-based access control (ABAC) [33]. CloudLens and CloudExploit primarily focus on RBAC policies, which are more commonly used in practice [7]. In RBAC, access to cloud resources is based on a role that usually aligns with the business logic [5]. ABAC policies determine access based on attributes of the subject, object, and can be converted to RBAC format [7].

An IAM policy is a set of permissions, attached to a cloud identity, that define what actions a user is allowed to execute on what AWS resources. A resource is a cloud object with a unique identifier, such as an S3 bucket (a directory in a distributed file storage system). Each policy consists of one or more statements and each statement describes a specific set of permissions.

A statement in an IAM policy includes the following key elements:

- Effect: It can be either 'Allow" or 'Deny." The 'Allow" effect grants the specified permissions, while the 'Deny" effect explicitly denies the specified permissions.
- Action: The specific AWS service or resource actions that the policy allows or denies. Actions can range from simple tasks like read (e.g., s3:GetObject ) to more complex operations involving multiple resources. These actions are also called permissions , which give the privilege of performing the associated operation. These actions are related to, but different from PDDL actions that we will define later.
- Resource: The AWS resources to which the actions apply. This can include specific ARNs (Amazon Resource Names).

The statement in Listing 1 allows the 's3:GetObject" action (which is the read permission) on the object with Amazon Resource Name (ARN) 'arn:aws:s3:dummyBucket' (which is a datastore). Every item of an IAM policy is allowed to be specified in the form of a regular expression that makes the policy able to grant multiple permissions to multiple cloud objects in a concise manner but also introduces security vulnerabilities because of making the policy over-permissive [21]. Furthermore, each IAM policy may include an optional condition element which acts as an additional filter, serving to further restrict the policy's scope.

Policies are associated with IAM identities, such as users , groups , or roles , and they determine what actions those identities are allowed or denied to perform on AWS resources. ## 3 CLOUDLENS: MODELING CLOUD IAM

Before attack graph and attack paths can be generated for vulnerability analysis, we need to represent different security permissions and relations among cloud identities, resources in a systematic manner. Our framework, CloudLens, provides such a representation. While graph based representation of IAM has been used earlier [28], they are limited in scope to analyzing privilege escalation attacks, and are not easy to translate into a automated planning framework.

## 3.1 Relational modeling of IAM

We consider users, roles, and groups as IAM identities. For simplicity, we consider datastores as IAM resources (it is possible to consider additional resources such as compute, and network among others). At any point in time, the state of the IAM system is represented using the following 3- and 4-tuples:

- ⟨ id 1, perm , id 2 ⟩ : This tuple implies that identity id 1 has permission perm over id 2. E.g., this permission can be assumeRole , which implies id 1 can assume the role id 2 and gain all associated permissions of id 2. The id 2 can be another identity (user, role, or group).
- ⟨ id , perm , ds ⟩ : This tuple implies that identity id has permission perm over datastore ds . Permissions can be full\_control, read, write, delete, etc.
- ⟨ id , id 1, perm , id 2 ⟩ : There are special types of permissions that allow an identity id to add a 3-tuple 𝜏 = ⟨ id 1, perm , id 2 ⟩ to the IAM state. Under certain conditions, an action can be taken (e.g., when id is a compromised user) to add 𝜏 to the system state.
- ⟨ id , id 1, perm , ds ⟩ : Same interpretation as above; except that ds is a datastore.

Such a simple tuple-based formulation is enough to model the majority of cloud operations in AWS, GCP [32] to analyze its security vulnerabilities. When we say an id has permission perm , it would refer to the tuple ⟨ id , perm , id 1 ⟩ or ⟨ id , perm , ds ⟩ ( id 1 and ds would be clear from the context).

For a concrete example of a 4-tuple, consider the permission iam:AttachGroupPolicy 2 in Listing 5. Assume identity id has this permission. This permission allows id to attach a policy with ARN (Amazon resource name) PID to the group with ARN GID . We represent this as tuple ⟨ id , GID , hasPolicy , PID ⟩ .

## 3.2 IAM Attacks as Sequential Decision Making

In this section, we present a minimal formal model of IAM attack problem that highlights its key aspects relating to permission flows and stripping away implementation-specific details.

State definition. At any instant 𝑡 , the IAM state 𝑠 𝑡 is a set of tuples as noted in section 3.1. In addition, we also have predicates compromised ( id ) which are true for identities that are compromised (i.e., the attacker controls them).

Let the current state be 𝑠 . We define the following action types: permissionFlow ( id 2 , id 1 ) This action is applicable when permissions can flow from id2 to id1 given the current state 𝑠 .

Action precondition: Let the precondition for this action be isFlowActive ( id 2 , id 1 ) ⊆ 𝑠 . It can be defined based on the interpretation of different permissions in the specific cloud type (e.g., AWS, GCP). Also note that this action can be instantiated for all possible identities id 1, id 2.

Intuitively, permissionFlow ( 𝑗, 𝑖 ) is equivalent to identity 𝑖 assuming the role of identity 𝑗 , which leads to permission flow from 𝑗 to 𝑖 . In a particular attack plan, an identity 𝑖 can only assume one role. Thus, if action permissionFlow ( 𝑗, 𝑖 ) is taken, then no other action permissionFlow ( 𝑘, 𝑖 ) (and 𝑘 ≠ 𝑗 ) can be taken. This can be easily ensured by using a predicate inFlow ( i ) which is false initially and becomes true after an action permissionFlow ( 𝑗, 𝑖 ) is taken (details omitted).

## 3.3 Attack graph formulation

We can construct an attack graph , analogous to the planning graph used by the graph plan algorithm for classical planning problems [8]. Intuitively, the plan graph is a layered graph consisting of alternating levels of:

- State level - Represents tuples (as in section 3.1) that can be true at that level. At the initial level, only those tuples that are true are listed. These tuples are analogous to security conditions in an attack graph.
- Action level - Represents the actions that can be executed at that level based on the tuples (i.e., action preconditions) from the previous level. An action has a link from each of its precondition, and a link to its effect (as noted in previous section).
- Effect propagation - Each action contributes its effects (the tuples it adds or deletes) to the next state level.

There are additional concepts such as mutually exclusive ( mutex ) actions, tuples, and iteratively deepening the planning graph which are detailed in [8]. This graph is analogous to an attack graph where finding a path (a sequence of actions) until action enableAttack can be executed solves the attack generation problem.

## 4 CLOUDEXPLOIT: AUTOMATED PLANNING FOR ATTACK GENERATION

As noted in previous section, a key challenge is to identify a sequence of actions (e.g., cloud API calls) that can achieve a malicious objective (e.g., launching an impact attack). This task-generating an action sequence to reach a goal-naturally aligns with AI planning. The ability to generate such action sequences is crucial for verifying the existence of security vulnerabilities. While model checking is used in previous cloud security works [18], formal methods typically provide a binary answer regarding the presence of a safety property, and cannot easily handle action costs. Therefore, we formulate the planning problem in section 3.2 using the planning domain definition language (PDDL) [26], which has been used earlier also for network security and penetration testing [9, 16, 30, 37].

PDDL is a standardized language used to describe planning problems and domains in automated planning systems [26]. It provides a formal way to define the states, actions, and goals for planning tasks. It separates the description of a planning problem into two main parts: the domain (which includes predicates, actions, and their effects) and the problem (which specifies the initial state and goal conditions). For more details on PDDL syntax, we refer to [14].

Based on the simplified model in the previous section, we now describe specific implementation of different actions for Amazon AWS in PDDL. We first parse an organization's cloud setup to construct the relational model as in Section 3.1. We next show how the planner (or the attacker) uses such relations starting from a compromised user to make progress towards accomplishing an attack (as noted in Table 1). The key to gain permissions to enable an attack is via permission flow between identities as described next.

## 5 RESULTS

Our evaluation methodology is twofold, addressing both the verification of modeling accuracy and the identification of actual attacks. For verification, we employ the IAM Vulnerable benchmark [11] developed by Bishop Fox [12], a penetration testing firm. This benchmark comprises 31 attack scenarios associated with various AWS cloud services, particularly those vulnerable to misconfigurations potentially leading to privilege escalation attacks. The accuracy of the model is demonstrated through the application of CloudExploit across 14 distinct cloud configurations belonging to various commercial organizations. Details about the experimental setup are in supplementary. Anonymized datasets used in our evaluation together with CloudExploit's implementation will be publicly released.

## 5.1 IAM Vulnerable Benchmark

We conducted a thorough evaluation of our model's accuracy against IAM Vulnerable benchmark [11], specifically focusing on the attack scenarios relevant to our modeling efforts. Remarkably, CloudExploit successfully detected all 19 attacks (out of 31) that are within the scope of PDDL modeling. Our code base provides PDDL problem data files for this benchmark.

In terms of IAM policy modeling, CloudExploit demonstrates comprehensive support for the most frequently used services, such as IAM, EC2, S3, Lambda, STS, and SSM. However, we deliberately chose to exclude less common services like SageMaker, CloudFormation, CodeBuild, and DataPipeline from our modeling process to keep PDDL problem files compact for solver scalability. This exclusion accounts for the 12 undetected test cases in the benchmark. In summary, our evaluation highlights CloudExploit's effectiveness in modeling and detecting vulnerabilities within the scope of widely used AWS services.

## 5.2 Real-world Datasets

CloudExploit is employed to analyze 14 commercial cloud configurations. Details of these cloud configurations are in the supplement. The data includes hundreds to thousands of identities and datastores, and several thousand permissions. To our knowledge, this dataset is one of the most extensive in cybersecurity for modeling cloud permission vulnerabilities.

To enable Fast Downward [15] solver to process such large datasets, we partition them based on AWS accounts that serve as effective boundaries for both administration and billing. In some cases account partitioning is not sufficient because the solver still runs out of memory. To handle those situations we partition an account into non-overlapping groups of users and include other types of identities and datastores that are reachable from the source users via permission edges using breadth-first search. We intentionally exclude admin users from our analysis, as compromising such users is likely to have a widespread impact on the entire system, attributable to their extensive array of potentially high-risk permissions.

Attack path diversity. Our evaluation methodology is designed to estimate the diversity of attack paths generated by CloudExploit. It is important to note that a mere syntactic analysis of these paths is insufficient, as their diversity could be artificially increased by the inclusion of irrelevant actions that do not alter their semantic meaning. For instance, a path that incorporates a role assumption action that is irrelevant to the attack may appear different from one without such a redundant element, yet both paths are semantically identical. Our approach to measuring attack diversity draws on principles from the cybersecurity domain. We specifically gauge the number of compromised users for each attack-data set combination. While there may exist other diverse paths, the critical factor for our analysis is the extent to which users are vulnerable to attacks.

Figure 1 presents the number of compromised users for each attack, data set combination. The y-axis indicates the count of compromised users, while the data sets are arrayed along the x-axis. CloudExploit identified 507 non-admin users across the datasets who could potentially be exploited by attackers to initiate one or more attacks if compromised.

A noticeable trend is the vulnerability of almost all real data sets to ransomware and impact attacks, such as encrypting and deleting sensitive data or removing critical cloud resources: identities, access keys essential for programmatic access to cloud resources, and IAM policies. The latter action usually leads to the removal of accesscontrol constraints, potentially resulting in privilege escalation.

At least one user in nearly every environment can execute a ransomware attack. This vulnerability stems from flaws in the IAM model. To conduct a ransomware attack, as detailed in Section 2.1, a user requires just two permissions: s3:PutObject and kms:CreateKey. The former is a common 'write' permission, while the latter is often inadvertently assigned through blanket KMS access, neglecting the exclusion of hazardous permissions. Additionally, CloudExploit evaluates two critical S3 bucket configuration parameters: versioning and multi-factor authentication (MFA). The activation of bucket versioning enables the restoration of any version of objects stored in an S3 bucket. Meanwhile, MFA protection mandates extra authentication for object deletion in an S3 bucket, enhancing security against unauthorized actions. In our experiments, we found that both of those configuration settings are disabled sometimes, which is a prerequisite for a successful ransomware attack.

## 6 FUTURE WORK

Diverse plans can be useful to identify the weakest precondition(s) that enable such attacks, and remove them. We tested the diverse planner [20], but found that generated plans were not useful as the planner added benign actions (e.g., list files in a directory), while keeping the same core attack path. It can be interesting to develop methods for minimal diverse plans.

A fully automated approach to policy repair necessitates additional information, such as the critical services required by different users, to ensure that essential accesses are not inadvertently revoked during the repair operation. We envision CloudExploit as a security co-pilot that IT administrators can utilize to assess the attack surface and proactively address potential policy misconfigurations.

## A LONG ATTACK PATH

To illustrate the effectiveness of our planning approach, we demonstrate a six-step attack (Listing 10) found in dataset #14. The attack begins with user\_9 who possesses permission to assume role role\_10 , which, in turn, can be used to assume role\_13 . Notably, role\_13 is endowed with dangerous permission iam:AttachRolePolicy that enables the execution of a broad spectrum of cloud actions. We model the iam:AttachRolePolicy permission as one that allows the attachment of adminPolicy , which comes with unrestricted permissions. Consequently, user\_9 can achieve unrestricted permissions through a sequence of role assumptions.

The attack unfolds in the following manner:

- selectCompromisedUser : The planner marks user\_9 as compromised.
- permissionFlow\_id\_4tpl : It establishes a permission flow from role\_13 to role\_10 using sts:AssumeRole , enabling the tuple ⟨ role\_10, role\_10, hasPolicy, adminPolicy ⟩ .
- permissionFlow\_id\_4tpl : It establishes a permission flow from role\_10 to user\_9 using sts:AssumeRole , enabling the tuple ⟨ user\_9, role\_10, hasPolicy, adminPolicy ⟩ .
- add\_id\_3tpl : A 3-tuple ⟨ role\_10, hasPolicy, adminPolicy ⟩ is added to the environment by the planner.
- permissionFlow\_id\_3tpl : The planner applies permission flow from role\_10 to the compromised user\_9 by enabling the tuple user\_9, hasPolicy, adminPolicy .
- reachAdminPolicy : user\_9 gains adminPolicy , providing broad permissions and enabling all attack types modeled by CloudExploit.

## D NP-HARDNESS: REDUCING SET COVER TO ATTACK PATH

Consider the set cover problem which is NP-Hard. We are given a set 𝑉 = { 𝑣 1 , . . . , 𝑣 𝑛 } of 𝑛 elements. We are also given 𝑚 subsets 𝑆 1 to 𝑆 𝑚 of 𝑉 . Goal is to find the minimum number of subsets whose union covers all elements in 𝑉 . We reduce this problem to our IAM attack path problem.

The set of identities is { 𝑆 } ∪ { 𝑆 1 , . . . , 𝑆 𝑚 } . We create an identity for each set 𝑆 𝑖 , and an additional identity 𝑆 .

We have 𝑛 datastores { 𝑣 1 , . . . , 𝑣 𝑛 } .

There are two permissions perm ∈ { canAssume , hasElement } . The intuition is that when we have tuples ⟨ 𝑆, ℎ𝑎𝑠𝐸𝑙𝑒𝑚𝑒𝑛𝑡, 𝑣 𝑖 ⟩∀ 𝑖 = 1 : 𝑛 , then it means that we have found a cover. And the cost of each action is 1, so only minimum number of subsets 𝑆 𝑗 will be used in the optimal attack plan.
