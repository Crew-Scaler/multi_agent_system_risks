# Risk Management Analysis: NIST AI Risk Management Framework

**Framework:** NIST AI Risk Management Framework (AI RMF 1.0) with companion documents
**Sources:**
- NIST AI 100-1 (2023): Artificial Intelligence Risk Management Framework (AI RMF 1.0) — core governance framework with four functions: GOVERN, MAP, MEASURE, MANAGE
- NIST AI 100-2e2025: Adversarial Machine Learning: A Taxonomy and Terminology of Attacks and Mitigations — attack taxonomy and technical mitigations for predictive and generative AI
- NIST SP 800-218A: Secure Software Development Practices for Generative AI and Dual-Use Foundation Models — secure development lifecycle practices

**Framework Summary:** The NIST AI RMF 1.0 provides a voluntary, flexible framework for managing AI risks throughout the AI system lifecycle. Its four core functions are: (1) GOVERN — establishing organizational policies, accountability structures, workforce capabilities, risk culture, stakeholder engagement processes, and supply chain risk management (subcategories GOVERN 1–6); (2) MAP — identifying context, categorizing AI systems, characterizing capabilities and limitations, mapping third-party risks, and assessing potential impacts (subcategories MAP 1–5); (3) MEASURE — employing quantitative and qualitative TEVV (test, evaluation, verification, and validation) methods to assess trustworthy characteristics, track risks over time, and gather measurement feedback (subcategories MEASURE 1–4); and (4) MANAGE — allocating resources to mapped and measured risks, planning response/recovery, managing third-party risks, and maintaining post-deployment monitoring (subcategories MANAGE 1–4). The companion document NIST AI 100-2 extends the framework with a detailed adversarial ML taxonomy covering evasion attacks, poisoning attacks (availability, targeted, backdoor, model), privacy attacks (data reconstruction, membership inference, model extraction), and generative AI-specific threats (supply chain, direct prompting/jailbreaking, indirect prompt injection) along with concrete mitigations. NIST SP 800-218A adds a secure software development community profile for AI model development. Together these documents address both governance and technical security aspects of AI risk, making this a notably more cybersecurity-relevant framework than pure governance frameworks. It does not prescribe specific infrastructure security controls (e.g., Kubernetes hardening, GPU isolation), does not address hardware-level attacks in detail, and its guidance on multi-agent agentic AI architectures is nascent.

## RATC - Agent–tool coupling as "policy‑level remote code execution"

### RATC_1 - UI/UX Abstraction and Visibility Gaps - Strength score: 2

RATC_1 threats center on UI/UX design patterns (abstraction layers, progressive disclosure, inline suggestions, multi-agent dashboards, trace visualization) that hide true tool chain complexity and risk levels from human approvers. NIST AI 100-1 GOVERN 2 establishes accountability structures requiring that appropriate teams are empowered and trained for AI risk management, which encompasses human oversight interface design. MAP 5 (impacts characterization) requires documenting the likelihood and magnitude of harmful impacts including those arising from inadequate human visibility into system operations. MEASURE 2.5 requires evaluating AI systems for human-AI interaction and oversight adequacy. MANAGE 4.1 mandates post-deployment monitoring that captures user feedback and incident signals. These framework elements collectively create organizational accountability for ensuring human reviewers have adequate visibility into tool invocations. However, NIST AI 100-1 provides no specific technical guidance on UI/UX design standards for multi-agent tool chain transparency, risk differentiation in dashboards, or progressive disclosure security requirements. The practical impact is governance-level pressure for adequate human oversight design without prescriptive technical requirements.

### RATC_2 - Approval Workflow and User Attention Vulnerabilities - Strength score: 2

RATC_2 threats include approval fatigue from excessive multi-agent decision volume, keyboard shortcut hijacking through browser extension exploitation, and time-based default exploits forcing approval through timeout. NIST AI 100-1 GOVERN 2.1 establishes clear roles, responsibilities, and accountability for those managing AI risk, encompassing oversight workflow design. MAP 5.1 requires characterizing the likelihood and magnitude of harmful impacts, creating grounds to assess approval workflow security as part of impact analysis. MEASURE 2.6 requires evaluating safety risks and demonstrating that residual negative risk does not exceed acceptable thresholds, which can be interpreted to include risks from exploitable human oversight failures. GOVERN 4.1 requires fostering a safety-first mindset in organizational practices. These elements provide governance foundations for requiring well-designed human oversight workflows. However, NIST AI 100-2 does not specifically address keyboard shortcut hijacking, focus manipulation, or timeout-based approval bypass as adversarial ML attack vectors. The framework's contribution is accountability-level governance rather than specific technical countermeasures.

### RATC_3 - Tool Overload, Parameter Handling, and Selection Errors - Strength score: 1

RATC_3 threats involve tool overload degrading multi-agent selection accuracy and enabling privilege escalation through hierarchical delegation, adversarial tool argument injection across agent trust boundaries exploiting probabilistic LLM parameter generation, inline suggestion tools executing invisibly, and chat interfaces obscuring tool provenance. NIST AI 100-1 MAP 2.1 requires defining tasks and methods the AI system supports, and MAP 2.2 addresses AI system knowledge limits and output uncertainties. MEASURE 2.1 requires documenting test sets and metrics for TEVV. These documentation and assessment requirements could conceptually encompass testing tool selection robustness and argument handling. However, the NIST AI RMF provides no specific guidance on adversarial parameter injection across multi-agent trust boundaries, privilege escalation through tool overload in hierarchical delegation, or the security implications of separating argument construction from execution across agent boundaries. The framework is insufficiently specific for these highly technical attack vectors.

### RATC_4 - Confidence Manipulation and Tool Authorization - Strength score: 1

RATC_4 describes adversarial prompt injection inflating agent confidence scores to bypass confidence-based authorization gates, multi-agent cascading trust failures when agents accept unvalidated confidence scores from peer agents, and authorization scope confusion enabling privilege escalation through compromised intermediate coordination agents. NIST AI 100-1 MAP 2.2 requires documenting AI system knowledge limits and how system outputs should be interpreted, which is conceptually relevant to confidence score reliability. MEASURE 2.6 requires evaluating safety risks including residual negative risk assessment. GOVERN 1.6 (in the underlying practices) requires risk management processes that systematically identify and analyze risks. NIST AI 100-2 does not specifically address confidence score manipulation as an adversarial attack vector distinct from general prompt injection. The framework provides no concrete guidance on adversarial confidence inflation, multi-agent transitive trust exploitation, or precondition validation bypass across delegation chains, making its contribution insufficient for these specific attack classes.

### RATC_6 - Tool Metadata Poisoning Across Registries and Discovery - Strength score: 2

RATC_6 threats involve poisoning centralized tool registries to affect all agents simultaneously through vector database metadata corruption, few-shot parameter injection in tool descriptions, and framework-specific tool selection manipulation. NIST AI 100-1 GOVERN 6.1 explicitly addresses policies and procedures for AI risks associated with third-party entities, which encompasses tool registry vendors and shared infrastructure. MAP 4.1 requires approaches for mapping legal and technical risks of AI system components including third-party software. MANAGE 3.1 requires regularly monitoring third-party AI risks and applying risk controls. NIST AI 100-2 §3.2 (supply chain attacks and mitigations) directly addresses data and model poisoning through third-party dependencies, with mitigations including cryptographic verification of downloads and data filtering — principles applicable to tool registry integrity verification. These elements together provide meaningful coverage of supply chain and third-party risk relevant to tool registry poisoning. The gap is in specific technical guidance for vector database metadata poisoning and few-shot injection in tool descriptions.

### RATC_8 - Advanced Tool Invocation Patterns and Inference - Strength score: 2

RATC_8 covers dynamic replanning exploitation through controlled failure injection, tool result injection in ReAct observation pipelines, and cross-framework schema translation vulnerabilities. NIST AI 100-2 §3.4 (indirect prompt injection attacks and mitigations) is directly relevant: it covers integrity attacks where malicious resources cause agents to perform attacker-specified tasks, knowledge base poisoning, execution triggers persisting through processing pipelines, and injection hiding via multi-stage attacks. The mitigations in §3.4.4 — including training for hierarchical trust relationships, detection schemes, input processing to filter third-party instructions, and spotlighting techniques — address the core mechanism of tool result injection in multi-agent pipelines. MEASURE 2.6 and MANAGE 4 provide governance structures for safety evaluation and post-deployment monitoring. The framework is less specific about multi-agent replanning security and cross-framework schema boundary protection, but the indirect prompt injection taxonomy offers applicable guidance.

### RATC_9 - Web and Multimodal Tool Exploitation - Strength score: 2

RATC_9 threats include adversarial instruction injection into web content propagating through multi-agent scraping pipelines, hidden form fields embedding agent instructions, and cross-modal parameter injection in multimodal RAG pipelines. NIST AI 100-2 §3.4 (indirect prompt injection) directly covers agent hijacking through manipulation of external resources, with integrity attacks where malicious web content causes agents to perform attacker-specified tasks. The mitigations in §3.4.4 cover filtering instructions from third-party data sources, spotlighting to separate trusted from untrusted data, and detection schemes — directly applicable to web content injection and multimodal pipeline attacks. MAP 4.1 requires mapping risks of third-party data sources. GOVERN 6 addresses supply chain risk from external data. The framework's coverage of web-based indirect injection is meaningful, though it does not specifically address multimodal attack surface boundaries or vision model output laundering in multi-step processing pipelines.

### RATC_10 - Efficiency Optimization and Resource Constraints - Strength score: 1

RATC_10 threats exploit token budget constraints to force upstream agents into contexts that truncate safety-critical tool descriptions — causing downstream agents to inherit abbreviated specifications — and exploit temperature/iteration budget variations to create exploitable policy inconsistency across agent fleets. NIST AI 100-1 MAP 2.2 requires documenting AI system knowledge limits including how outputs may degrade, and MEASURE 1.1 requires selecting measurement approaches for significant AI risks. However, the NIST AI RMF provides no specific guidance on adversarial context window manipulation, token budget exploitation to truncate safety context, or temperature tuning diversity creating policy inconsistency. NIST AI 100-2 notes that inference parameter restrictions (e.g., limiting temperature access) can reduce attack efficacy as a deployment-time mitigation, but this is a brief mention rather than comprehensive guidance. The framework does not adequately address resource constraint exploitation as an attack class.

### RATC_11 - Tool Execution Infrastructure and Orchestration - Strength score: 2

RATC_11 describes API gateway routing manipulation redirecting tool calls to attacker-controlled services and Kubernetes/container orchestration attacks enabling privilege escalation and fleet-wide compromise. NIST AI 100-1 MAP 4.1 requires approaches for mapping technical risks of AI system components including third-party software and infrastructure. GOVERN 6 addresses policies for supply chain risks from third-party entities. MANAGE 3.1 requires monitoring third-party risks and applying controls. NIST SP 800-218A (SSDF for GenAI) includes secure development practices covering vulnerability management and dependency security relevant to container supply chain integrity. These elements provide governance foundations for infrastructure risk management. However, NIST AI 100-1 does not prescribe specific API gateway security controls, container security context hardening, Kubernetes RBAC configurations, or PVC security policies — detailed infrastructure security guidance is outside the framework's intended scope.

### RATC_12 - Distributed and Hardware-Level Attacks - Strength score: 1

RATC_12 covers tensor parallelism inter-GPU communication interception enabling tool-calling policy corruption, KV cache poisoning in shared multi-user inference enabling cross-session tool bias, and quantization calibration dataset poisoning biasing downstream parameter outputs. NIST AI 100-1 MAP 4.1 requires mapping technical risks of AI system components, and MEASURE 2.6 requires evaluating safety risks. NIST AI 100-2 §2.3 covers poisoning attacks broadly with mitigations including data filtering and integrity checks. However, the NIST AI RMF provides no specific guidance on GPU communication security (encrypting inter-GPU traffic in tensor parallelism), KV cache isolation in multi-tenant inference, MIG instance security, or hardware-level attack surface management. These attack vectors require hardware and infrastructure-level defenses beyond the scope of the AI RMF's governance and ML security guidance.

### RATC_14 - Reasoning and Planning Vulnerabilities - Strength score: 1

RATC_14 covers chain-of-thought reasoning manipulation enabling embedded justifications for dangerous tool sequences, self-consistency and majority voting manipulation through sampling path convergence, and hierarchical task network planning fragmentation creating tool visibility gaps between planning and execution layers. NIST AI 100-1 MAP 2.2 requires documenting knowledge limits and how system outputs should be interpreted, and MEASURE 2.6 requires safety risk evaluation. NIST AI 100-2's taxonomy covers prompt injection and jailbreaking but does not specifically address adversarial chain-of-thought reasoning manipulation, majority voting confidence inflation, or planning hierarchy exploitation as distinct attack classes. The framework provides insufficient specific guidance for these reasoning and planning exploitation vectors targeting internal LLM decision processes.

### RATC_15 - Episodic Memory and Learning - Strength score: 2

RATC_15 covers poisoning of episodic memory storing tool action records so agents retrieving prior "successful" episodes replicate malicious tool chains, and multi-agent shared episodic memory enabling compromise propagation across the agent fleet. NIST AI 100-2 §2.3 (poisoning attacks and mitigations) directly addresses data poisoning of training pipelines with mitigations including data filtering and integrity verification — principles applicable to episodic memory stores. §2.3.2 (targeted poisoning) and §2.3.3 (backdoor poisoning) cover scenarios where specific triggering patterns cause malicious behavior, relevant to trajectory-level poisoning. MAP 4.1 requires mapping risks of data pipeline components. MEASURE 3.1 requires tracking risks over time including emergent risks from evolving system behavior. These elements provide meaningful but incomplete coverage: the AML taxonomy addresses the poisoning mechanism but does not specifically discuss episodic memory architectures in agentic AI systems.

### RATC_16 - Semantic Memory and RAG - Strength score: 3

RATC_16 covers RAG pipeline and knowledge base poisoning where attackers embed tool-invocation instructions in retrieved content, knowledge graph relationship poisoning causing unsafe tool combinations to appear recommended, query rewriting instruction injection, and knowledge base staleness causing obsolete tool invocations. NIST AI 100-2 §3.4.2 specifically addresses knowledge base poisoning of RAG systems (citing PoisonedRAG and Phantom attacks), execution triggers persisting through RAG processing pipelines, and injection hiding techniques — a direct and detailed match to RATC_16 threats. §3.4.4 provides concrete mitigations: fine-tuning for hierarchical trust, detection schemes, filtering instructions from third-party data sources, spotlighting, and instructing models to disregard instructions in untrusted data. MAP 4.1 and GOVERN 6 address third-party data source risk management. This combination of specific attack identification and concrete mitigation guidance in 100-2 directly and strongly supports management of RAG-based semantic memory poisoning threats.

### RATC_17 - Utility Functions and Decision Logic - Strength score: 1

RATC_17 covers expected utility calculation vulnerabilities including outcome assumption injection causing miscalculation of tool utility, sequential expected utility miscalculation misrepresenting future options, and trade-off weight manipulation poisoning multi-objective utility functions. NIST AI 100-1 MAP 2.2 requires documenting knowledge limits and output interpretation, and MEASURE 2 requires evaluating trustworthy characteristics. However, neither the AI RMF nor the AML taxonomy in 100-2 specifically addresses adversarial manipulation of utility function parameters, multi-objective trade-off weight poisoning, or sequential utility miscalculation as distinct attack vectors. These threats target the internal decision logic of planning agents, which falls outside the scope of the framework's current guidance.

### RATC_18 - Rule-Based and Knowledge-Engineered Systems - Strength score: 1

RATC_18 covers rule-based authorization bypass through specificity hierarchy exploitation injecting more-specific rules overriding safety rules, forward chaining exploitation via fact injection triggering tool authorization chains, and certainty factor manipulation exploiting high-confidence trust to bypass risk assessment. NIST AI 100-1 GOVERN 1 requires policies and processes for managing AI risks and MAP 2.1 requires defining AI system tasks and methods. However, neither NIST AI 100-1 nor 100-2 provides specific guidance on rule-based system security, forward chaining rule chain exploitation, certainty factor manipulation, or lexicographic heuristic objective reordering in knowledge-engineered systems. These are specialized attacks on classical AI planning architectures that fall outside the scope of the AI RMF's ML-focused security taxonomy.

### RATC_19 - Learning and Reinforcement Learning - Strength score: 2

RATC_19 covers RL tool selection policy hijacking through reward poisoning training agents toward malicious tools, multi-agent RL shared replay buffer poisoning affecting all agents' Q-values, curriculum learning poisoning, imitation learning trajectory poisoning, actor-critic cross-agent critic poisoning, and PPO trust region manipulation. NIST AI 100-2 §2.3 (poisoning attacks and mitigations) addresses data poisoning across training pipelines including reinforcement learning from human feedback (§3.2.1), with mitigations including data filtering and integrity verification. §2.3.3 (backdoor poisoning) covers scenarios where training-time manipulation causes specific triggering conditions to activate malicious behavior, directly relevant to reward signal manipulation. MEASURE 3.1 requires tracking risks from evolving AI system behavior. MAP 2.3 addresses scientific integrity in data collection relevant to human feedback pipelines. The framework provides meaningful coverage of the poisoning mechanism, though it does not specifically address multi-agent shared replay buffer architecture exploitation or PPO trust region manipulation as distinct attack vectors.

### RATC_21 - Parallel Retrieval Race Conditions in Multi-Agent Query Decomposition Systems - Strength score: 1

RATC_21 covers parallel retrieval race conditions creating inconsistent results across concurrent sub-queries, shared index deduplication races, timeout-triggered cascades overwhelming connection pools, multi-stage caching temporal inconsistency from varying TTL expiration, container supply chain vulnerabilities through registry compromise, and centralized inference infrastructure single points of failure. NIST AI 100-1 MEASURE 3.1 requires tracking emergent risks and MANAGE 4.1 mandates post-deployment monitoring. MAP 4.1 addresses third-party software risks. However, the NIST AI RMF provides no specific guidance on distributed retrieval system race conditions, TTL-based consistency attacks, cache poisoning via hash collision, container registry supply chain security for AI deployments, or tensor parallelism failure propagation. While GOVERN 6 addresses supply chain risk broadly, the framework does not address the concurrent execution vulnerabilities and infrastructure failure cascade patterns described in RATC_21.

### RATC_22 - Multi-Stage Pipeline Result Caching Race Enabling Multi-Agent Consistency Failures - Strength score: 1

RATC_22 covers multi-stage pipeline caching race conditions with independent TTL expiration causing temporal inconsistency windows, event-based cache invalidation propagation creating fleet-wide inconsistency, shared caching write races, adversarial hash collision cache poisoning, GitOps automation enabling fleet-wide backdoor deployment, and tensor parallelism communication poisoning through multi-tenant collocation. NIST AI 100-2 §3.2 (supply chain attacks) covers container registry compromise and model poisoning through third-party model supply chains, with verification and integrity checking mitigations applicable to container image integrity. MANAGE 3.1 addresses third-party risk monitoring. However, the framework does not specifically address distributed cache consistency attacks, hash collision cache poisoning vectors, GitOps automation security, or tensor parallelism vulnerability in multi-tenant deployments as distinct risk categories warranting specific controls.

### RATC_27 - MIG Instance Co-Location Creating Shared Physical GPU Failure Propagation - Strength score: 1

RATC_27 covers Multi-Instance GPU (MIG) logical isolation limitations where shared physical components (power regulators, thermal management, PCIe interface, GPU firmware, driver stack) create fleet-wide failure propagation risk despite MIG partitioning, and firmware vulnerabilities enabling cross-partition memory access breaking isolation guarantees. NIST AI 100-1 MAP 4.1 requires mapping technical risks of AI system components. MEASURE 2.6 requires safety risk evaluation. However, neither NIST AI 100-1 nor 100-2 addresses GPU hardware isolation security, MIG partition security boundaries, CUDA driver stack vulnerability management, or physical layer dependency risks in AI inference infrastructure. These are hardware and infrastructure-level concerns outside the scope of the AI RMF's current guidance.

### RATC_29 - Shared Workflow State Version Conflict Amplification Through Concurrent Write Flooding - Strength score: 1

RATC_29 covers optimistic locking version conflicts in shared multi-agent workflow state where high concurrency creates systematic version conflicts, retry storms that intensify contention through positive feedback loops, and targeted collision bursts exhausting retry budgets to cause workflow failures. NIST AI 100-1 MEASURE 3.1 requires tracking risks over time and MANAGE 4.1 requires post-deployment monitoring. MAP 5.1 addresses characterizing likelihood and magnitude of harmful impacts. However, the NIST AI RMF provides no specific guidance on distributed concurrency control security, optimistic locking exploitation, retry storm attack patterns, or workflow state consistency under adversarial concurrency flooding. These are distributed systems engineering concerns that the AI RMF is not designed to address.

### RATC_30 - Guardrail Bypass Through Infrastructure Failure Injection - Strength score: 2

RATC_30 covers adversary-induced processing delays causing guardrail validation timeouts and fleet-wide safety bypass, service unavailability attacks through network flooding or resource exhaustion against centralized validation endpoints, and fail-open configurations delivering unvalidated responses when safety infrastructure is unavailable. NIST AI 100-1 MEASURE 2.6 requires demonstrating that residual negative risk does not exceed acceptable thresholds and evaluating safety risks — directly applicable to assessing guardrail reliability under adversarial conditions. MANAGE 1.1 requires determining whether AI systems achieve their intended purposes, including safety guardrails. MANAGE 4.1 mandates monitoring that captures safety incidents. GOVERN 1.5 requires ongoing monitoring and periodic review of risk management outcomes. NIST AI 100-2's deployment-time mitigation recommendations include monitoring and response to detect and remediate safety failures. These governance and measurement elements support identifying and planning for guardrail failure scenarios, though the framework provides no specific technical guidance on fail-safe versus fail-open configurations, validation infrastructure redundancy, or timeout exploitation defense.

## RDL - New data‑leakage channels via large contexts, logs, and probabilistic recall
### RDL_1 - UI/UX Patterns - Strength score: 2

RDL_1 covers a broad set of UI/UX data leakage risks: chat interface conversation histories accumulating multi-agent sensitive data without access controls, progressive disclosure debug views exposing credentials and authentication tokens, session persistence crossing security boundaries, command palette recent history logging sensitive operations, multi-agent dashboards correlating data across compartmentalized sources, reference links enabling indirect information disclosure, approval workflow audit trails as comprehensive behavior logs, supporting evidence links as aggregation points, refinement iteration histories exposing decision logic, and ARIA/alt text exposing sensitive values to screen readers. NIST AI 100-1 MAP 5.1 requires characterizing likelihood and magnitude of harmful impacts from AI system deployment, which encompasses exposure from inadequate UI access controls. GOVERN 2 establishes accountability structures for those responsible for AI risk management. MEASURE 2.5 addresses human-AI interaction adequacy. However, NIST AI 100-2's privacy attack taxonomy (§2.4, §3.3.2) covers information extraction and context-stealing attacks, providing relevant adversarial framing for these UI-mediated leakage risks. The framework's GOVERN and MAP functions create organizational accountability for identifying and managing these data exposure risks, though it provides no specific UI/UX security design guidance or multi-agent dashboard security requirements.

### RDL_2 - Data Persistence and Caching - Strength score: 1

RDL_2 covers session persistence serialization converting sensitive conversation into queryable structured records accessible beyond session lifetime, performance caching creating persistent sensitive copies in browser and CDN layers with weaker access controls, and undo functionality maintaining recoverable copies of data users believe deleted in violation of GDPR right-to-be-forgotten. NIST AI 100-1 MAP 4.1 requires mapping technical risks of AI system components, and GOVERN 6 addresses third-party data risks. MEASURE 2.6 requires safety risk evaluation. However, the NIST AI RMF provides no specific guidance on session storage security, cache access control requirements, undo buffer security in distributed multi-agent systems, or data residency and deletion compliance across agent boundaries. These are application-level data engineering concerns that the framework's governance and ML security guidance does not specifically address.

### RDL_3 - Streaming and Token-Level Leakage - Strength score: 1

RDL_3 covers fine-grained leakage through streaming token display before redaction can apply, tokenization pattern inference enabling context window mapping, output length correlation enabling session reconstruction without content access, streaming state updates in iterative cycles exposing intermediate states, intermediate states revealed through batch/load-balanced streaming, and streaming cache side channels enabling data recovery through cache metadata patterns. NIST AI 100-1 MEASURE 3.1 requires tracking emergent risks and MANAGE 4.1 mandates monitoring. However, neither NIST AI 100-1 nor 100-2 specifically addresses streaming as a distinct attack surface — token-by-token leakage timing, streaming cache side channels, output length inference attacks, or tokenization pattern exploitation are not covered by the framework's attack taxonomy or mitigation guidance. These are application-level streaming architecture security concerns outside the scope of the AI RMF.

### RDL_4 - Search and Recall - Strength score: 2

RDL_4 covers probabilistic recall through semantic conversation search where queries surface semantically related sensitive content across multi-agent histories without security boundary enforcement, enabling systematic probing to surface high-sensitivity content through semantic association. NIST AI 100-2 §3.3.2 directly addresses information extraction attacks where sensitive information in RAG databases or context is leaked through query access, and §3.4.3 covers privacy compromise through indirect injection causing agents to leak restricted information. These privacy attack taxonomies directly apply to semantic search enabling information extraction across multi-agent conversation stores. MAP 4.1 requires mapping risks of data pipeline components. MEASURE 2.3 addresses data collection and use. The framework provides meaningful adversarial framing for semantic-search-enabled privacy attacks, though it does not specifically address multi-agent search index boundary enforcement or policy-controlled semantic search as distinct requirements.

### RDL_5 - Attribution and Observability - Strength score: 1

RDL_5 covers attribution metadata logs exposing organizational structure and high-privilege user patterns, framework-dependent logging aggregating multi-framework telemetry into comprehensive system architecture disclosures, error classification patterns revealing infrastructure fragility, and latency measurement enabling attack timing inference. NIST AI 100-1 MANAGE 4.1 requires post-deployment monitoring, and MEASURE 3.1 requires tracking risks from deployed systems. These monitoring requirements support the existence of logging infrastructure but do not address its security. The NIST AI RMF provides no specific guidance on log access control, attribution metadata minimization, cross-framework log aggregation security, or side-channel protection for error and latency telemetry. These are security observability engineering concerns beyond the framework's scope.

### RDL_7 - Tool Invocation and Function Calling - Strength score: 1

RDL_7 covers tool invocation logs as side channels exposing parameters containing sensitive data, function calling JSON persisting across agent boundaries in conversation history, detailed error messages leaking implementation internals, tool selection reasoning stored in memory exposing decision logic, function description enumeration revealing organizational capabilities, audit trail poisoning where compromised agents omit malicious invocations, tool success rate differential leakage enabling agent compromise targeting, parameter value leakage through centralized action logs, accuracy metrics exposing operational patterns, and log aggregation concentrating sensitive data from all agents. These are application security engineering concerns not covered by the AI RMF's current guidance.

### RDL_9 - Multi-Agent Memory and State - Strength score: 2

RDL_9 covers cross-agent reflection memory becoming unauthorized conduits for sensitive data across trust boundaries, ReAct reasoning trace persistence creating comprehensive attack graphs, distributed trace cross-tenant correlation enabling workflow inference through timing metadata, blackboard shared memory access pattern analysis enabling workflow reconstruction, and multi-agent dashboard correlation leakage from simultaneous agent activity patterns. ### RDL_10 - Reasoning and CoT Traces - Strength score: 2

### RDL_11 - Multimodal and Input Processing - Strength score: 2

MAP 4.1 addresses risks of third-party components including multimodal processing pipelines. These are application security engineering concerns beyond the framework's scope.

### RDL_13 - Evaluation and Testing Leakage - Strength score: 2

RDL_13 covers evaluation pipeline metric logs creating system fingerprints, baseline comparison agent context windows exposing historical test data, evaluation dashboard test case sample exposure, distributed evaluation artifact storage leakage, comprehensive evaluation script logging enabling evasion, metric output fingerprinting enabling customized manipulation, temporal evaluation result analysis inferring model update patterns, cross-validation dataset structure leakage, and benchmark result approval logs exposing agent failure modes. MEASURE 4.2 addresses measurement efficacy, including the integrity of evaluation processes. ### RDL_15 - Evaluation Metric Exploitation - Strength score: 2

RDL_15 covers exact match metric exploitation through semantic paraphrasing bypassing correct-answer detection, joint metric evasion through fabricated supporting facts optimized for metric satisfaction, Pass@K inconsistency exploitation creating probabilistic attacks invisible to aggregated metrics, and milestone scoring threshold manipulation enabling conditional execution targeting partial credit boundaries. MAP 2.3 addresses scientific integrity and TEVV considerations including experimental design. GOVERN 4.1 requires critical thinking and safety-first approaches to AI risk. The MEASURE framework's emphasis on rigorous, scientifically valid, and manipulation-resistant evaluation metrics provides governance accountability for metric integrity. NIST AI 100-2 also notes that evaluation can identify model vulnerabilities. While the framework does not specifically address adversarial metric gaming, its rigor requirements for TEVV create meaningful accountability for metric manipulation risks.

### RDL_16 - Parameter and Configuration Leakage - Strength score: 1
