# Risk Management Analysis: DoD AI Cybersecurity Risk Management Tailoring Guide

**Framework:** DoD Artificial Intelligence Cybersecurity Risk Management Tailoring Guide, Version 2, 14 July 2025
**Source:** `DoD_AI_Cyber_RMF/AI-CybersecurityRMTailoringGuide.pdf`

## Framework Summary

The DoD AI Cybersecurity RMF Tailoring Guide implements DoDI 8510.01 (Risk Management Framework for DoD Systems) as applied to AI systems throughout their lifecycle. It extends NIST SP 800-37, CNSSI 1253, and NIST AI RMF 1.0 to AI-specific contexts, covering all AI systems operated by or on behalf of DoD. The framework is organized around the AI system lifecycle:

- **Design and Develop**: Supply chain integrity (SA-4, SR-2/3/4/6/9), risk assessment (RA-3, PM-9/11), acquisition controls, hardware/software provenance.
- **Infrastructure Layer Development**: Access control (AC family), identification/authentication (IA), audit/accountability (AU), configuration management (CM), cryptographic protection (SC-8/13/28), integrity checking (SI-7).
- **AI Model Development**: DevSecOps process, TEVV (Test, Evaluation, Verification, Validation), software integrity (SI-7), static code analysis (SA-11(1)), SCRM (SR-8), auditing (AU-6), monitoring (SI-4), data security plans, privacy assessments, extensive security engineering principles (SA-8, 33+ sub-controls).
- **AI System Deploy and Use**: All access control and authentication controls, monitoring (SI-4, CA-7), vulnerability scanning (RA-5), DoS protection (SC-5), boundary protection (SC-7), input validation (SI-10), output filtering (SI-15), memory protection (SI-16), tainting (SI-20), incident response (IR-4/5/8), contingency planning (CP-2/3/4/9).
- **AI System Monitoring**: Continuous monitoring (CA-7), trend analysis (CA-7(3)), incident monitoring (IR-5), vulnerability scanning (RA-5). Specifically addresses detecting deviations from expected behavior, chaff data spamming, cost harvesting, and adversarial queries.
- **AI System Decommissioning**: Sanitization and destruction of model weights, training data, containers, and associated infrastructure.

The framework uses MITRE ATLAS as its primary AI threat taxonomy and maps ATLAS-derived threat vectors to CNSSI 1253 controls. Key threat categories addressed include supply chain compromise, model poisoning (data injection/manipulation/logic corruption), inference attacks (model extraction, inversion, API exfiltration), evasion attacks, DoS, unauthorized access, and misconfiguration.

Key limitations: the framework is a traditional RMF-based compliance document not specifically designed for agentic AI architectures. Its threat model (derived from ATLAS classical ML threats) does not cover LLM-specific agentic threats such as prompt injection, multi-agent trust exploitation, tool-calling security, reasoning manipulation, or agent memory poisoning. It focuses on infrastructure and model-layer security rather than runtime agentic behavior.

## Scoring Key

- **3** — Framework directly and strongly supports management of the risks (specific applicable guidance).
- **2** — Framework indirectly and moderately supports management (relevant principles or controls that partially apply).
- **1** — Framework weakly supports management (only general principles with minimal direct applicability).

## RATC — Agent–tool coupling as "policy-level remote code execution"

### RATC_1 - UI/UX Abstraction and Visibility Gaps
- Strength score: 1

RATC_1 describes how approval UIs, progressive disclosure patterns, chat interfaces, and multi-agent dashboards obscure the true scope of tool invocations—collapsing dangerous multi-tool chains behind benign-looking descriptions and denying users the visibility needed to assess actual risk. The DoD AI Cyber RMF does not address agent UI/UX design, approval workflow transparency, or the problem of user-facing abstraction layers masking policy-level actions. The framework's AU (audit logging) family requires logging of system events, which would partially record tool executions after the fact, and SI-4 (monitoring) could detect anomalous tool invocation patterns at the infrastructure level. However, neither control addresses the pre-approval visibility gap—the core vulnerability described here. General SA-8 security engineering principles loosely encourage defense-in-depth design, but the framework contains no guidance on how tool invocation risk should be surfaced, differentiated, or communicated to human approvers in agentic systems. This class of threat is entirely absent from the framework's ATLAS-derived threat model.

### RATC_2 - Approval Workflow and User Attention Vulnerabilities
- Strength score: 1

RATC_2 covers approval fatigue from high-volume multi-agent decision routing, keyboard shortcut hijacking through browser-level injection, and time-based default exploits where timeout mechanics force unauthorized approvals during coverage gaps. The DoD AI Cyber RMF does not address human approval workflow security, approval fatigue, or UI-layer manipulation. AC-2 (account management) and IA controls govern who is authorized to approve actions but say nothing about the integrity or timing mechanics of approval interfaces. SC-7 (boundary protection) and SC-23 (session authenticity) address network-layer session integrity but not browser-based UI manipulation. The framework's incident response controls (IR-4) would apply after a bypass occurs, but there is no preventive control framework for the cognitive and interface vulnerabilities described here. These are agentic-runtime and human-factors risks outside the scope of the traditional RMF model.

### RATC_3 - Tool Overload, Parameter Handling, and Selection Errors
- Strength score: 1

RATC_3 describes how excessive tool counts degrade agent selection accuracy and create privilege escalation paths in hierarchical delegation chains, how LLM-probabilistic argument construction enables injection across trust boundaries, and how inline suggestion and chat UI patterns conceal the tool provenance of executed actions. The framework's SI-10 (input validation) applies to validating data inputs to systems and could loosely cover parameter sanitization at tool boundaries, and AC-6 (least privilege) is conceptually relevant to the privilege escalation scenario. However, neither control addresses LLM-driven argument construction errors, cross-agent trust boundary injection, or the tool overload degradation dynamic. The framework has no concept of tool selection error rates, agent cognitive load, or multi-agent delegation chains as security concerns. SI-15 (output filtering) and AU logging provide partial post-hoc coverage but do not address the root causes described.

### RATC_4 - Confidence Manipulation and Tool Authorization
- Strength score: 1

RATC_4 addresses two threats: adversarial inflation of agent confidence scores to bypass confidence-gated approval automation, and authorization scope confusion where compromised coordination agents leverage transitive trust to execute tools without re-verifying prerequisites. The framework has no concept of confidence scores, confidence-based gating, or how LLM uncertainty estimates factor into authorization decisions. AC-4 (information flow enforcement) and AC-5 (separation of duties) partially touch the authorization scope concern—particularly the idea that a coordination agent should not inherit all downstream agents' privileges—but the framework does not operationalize these controls for multi-agent trust delegation. The precondition validation bypass (one agent verifies, another blindly trusts) is conceptually related to trust hierarchies addressed in SA-8(10) (hierarchical trust), but that principle is articulated at a high level of abstraction with no agentic-specific implementation guidance.

### RATC_6 - Tool Metadata Poisoning Across Registries and Discovery
- Strength score: 2

RATC_6 describes poisoning of centralized tool registries—including vector database metadata, few-shot parameter examples embedded in tool descriptions, and framework-specific tool selection mechanisms—to manipulate how agents understand and invoke tools across a multi-agent fleet. The DoD AI Cyber RMF has meaningful, directly applicable coverage of supply chain and registry integrity. SR-3 (supply chain controls and plans), SR-4 (provenance), SR-6(1) (supply chain risk assessment), and SA-4 (acquisition requirements) collectively address the integrity of software and service dependencies. SI-7 (software/firmware integrity checking) applies to detecting unauthorized modifications to tool definitions or registry entries. CM-2/6 (baseline configuration management) would govern what tool configurations are authorized. The framework's supply chain threat model explicitly includes software component compromise, which maps to tool registry poisoning at the infrastructure layer. The gap is that the framework does not address the semantic layer of poisoning—malicious instructions embedded in natural-language tool descriptions or few-shot examples that manipulate LLM behavior—since this is an LLM-specific attack vector absent from the ATLAS classical ML threat model.

### RATC_8 - Advanced Tool Invocation Patterns and Inference
- Strength score: 1

RATC_8 covers three advanced attack patterns: exploitation of Plan-and-Execute replanning loops by inducing controlled failures that progressively chain innocuous tools into dangerous sequences across agents; tool result injection where one agent's output becomes another agent's poisoned observation in ReAct-pattern chains; and tool schema variation attacks at cross-framework boundaries where transformation gaps allow malicious parameters to pass through undetected. The framework's SI-10 (input validation) and SI-15 (output filtering) provide partial applicability to the observation injection scenario—an agent could be required to re-validate received tool outputs—but the framework does not address multi-agent observation chains or the ReAct pattern specifically. AU logging would record individual tool calls, and CA-7 (continuous monitoring) could detect anomalous sequences post-hoc. However, the replanning loop exploitation and cross-framework schema transformation attacks are agentic architecture-specific vulnerabilities with no corresponding control in the CNSSI 1253 catalog as applied by this framework.

### RATC_9 - Web and Multimodal Tool Exploitation
- Strength score: 1

RATC_9 describes web agents that receive injected instructions from adversarial web content (a form of indirect prompt injection propagating across a scraping pipeline) and multimodal RAG systems where poisoned image or document content drives malicious tool parameter construction across retrieval and synthesis agent boundaries. The framework's SI-10 (input validation) is the closest applicable control and could require sanitization of content retrieved from external web sources. SC-7 (boundary protection) addresses network-level boundaries but not semantic content-level injection. The supply chain controls (SR family) would govern the integrity of web-scraping tool components themselves but not the adversarial content those tools retrieve. Multimodal cross-modal injection—where parameters are assembled from contributions across multiple modalities without sanitization—is entirely absent from the framework's threat model. Prompt injection from web content, a primary mechanism here, is explicitly identified in the framework cache as a gap.

### RATC_10 - Efficiency Optimization and Resource Constraints
- Strength score: 1

RATC_10 describes how token budget constraints in multi-agent systems cause upstream agents to produce truncated tool descriptions and reasoning traces, enabling attackers to force upstream agents into constrained contexts that omit safety-critical information. Temperature and iteration budget variation across agent specializations creates policy inconsistency where identical attacks succeed against some agents and fail against others. The framework has no concept of token budget management, context window constraints, or how LLM inference parameters (temperature, iteration limits) create security-relevant behavioral inconsistency. SC-5 (DoS protection) and CP controls address resource exhaustion at the infrastructure level, which is tangentially related to forcing constrained contexts, but the specific threat of context truncation defeating safety descriptions is an LLM-specific mechanism with no framework analog. This category of threat is entirely outside the framework's ATLAS-derived threat model.

### RATC_11 - Tool Execution Infrastructure and Orchestration
- Strength score: 2

RATC_11 addresses API gateway routing policy manipulation to redirect tool calls to attacker-controlled or vulnerable backend services, and Kubernetes/container orchestration attacks including security context modification, PVC compromise, sidecar proxy interception, and malicious init container injection that affect the entire multi-agent fleet. The DoD AI Cyber RMF has meaningful coverage of infrastructure-layer threats. CM-2/5/6/7 (configuration management) directly applies to API gateway policy and container security context configurations. AC-4 (information flow enforcement) and SC-7 (boundary protection) govern routing and network segmentation. SR-9 (supply chain risk—anti-counterfeit/tamper) and SI-7 (integrity checking) apply to container image and init configuration integrity. The framework explicitly covers infrastructure misconfiguration as a threat vector. The gap is that the framework does not specifically address AI gateway patterns, multi-agent microservice orchestration, or the particular risk that a single routing policy change compromises all agents simultaneously—but the underlying infrastructure controls are applicable and directly relevant.

### RATC_12 - Distributed and Hardware-Level Attacks
- Strength score: 1

RATC_12 covers two GPU-level attacks: interception and corruption of unencrypted inter-GPU tensor parallelism communication to hijack tool-calling policy decisions mid-computation, and KV cache poisoning in multi-user inference environments to systematically bias tool selection through fabricated attention values, as well as quantization calibration dataset poisoning. The framework cache explicitly identifies GPU-level hardware attacks (tensor parallelism, KV cache) as a gap not covered by the DoD AI Cyber RMF. SC-8 (transmission confidentiality and integrity) could theoretically apply to inter-GPU communication if treated as a network channel, and SC-13 (cryptographic protection) would address encrypting that channel, but the framework does not contemplate these GPU-internal communication paths. PE (physical/environmental) controls govern physical access to hardware. The KV cache sharing and quantization poisoning threats are novel hardware-ML system intersections entirely outside the ATLAS threat taxonomy used by the framework.

### RATC_14 - Reasoning and Planning Vulnerabilities
- Strength score: 1

RATC_14 describes tool selection vulnerabilities arising from chain-of-thought reasoning quality failures where poisoned reasoning traces embed justifications for dangerous tool sequences, self-consistency majority voting exploitation where all sampled reasoning paths are manipulated to converge on malicious tools with artificially high apparent confidence, and hierarchical task network planning fragmentation where high-level planning lacks tool awareness. The framework does not address LLM reasoning chains, chain-of-thought security, or self-consistency sampling as security mechanisms. The framework's TEVV (Test, Evaluation, Verification, Validation) process in Phase 3 would include testing model behavior, which loosely relates to verifying reasoning quality, and SA-11(2) (threat modeling) could theoretically encompass reasoning manipulation in a threat model. However, the framework has no controls targeting runtime reasoning integrity, majority voting exploitation, or planning-execution context fragmentation. Reasoning manipulation is explicitly identified in the framework cache as outside the framework's scope.

### RATC_15 - Episodic Memory and Learning
- Strength score: 1

RATC_15 describes poisoning of episodic memory stores where attackers record malicious tool sequences as "successful resolutions," causing agents that retrieve similar episodes to replicate the malicious chains, with cross-agent propagation through shared episodic memory enabling fleet-wide compromise. The framework does not address agent memory systems, episodic memory stores, or trajectory-based learning. SI-7 (integrity checking) could theoretically apply to detecting unauthorized modification of a memory database treated as a data store, and AC controls would govern access to such a store. However, the semantic poisoning attack—writing plausibly legitimate-looking malicious episodes—would not be detected by integrity checks targeting unauthorized modification. The framework's data integrity controls do not extend to the semantic validity of learned trajectories. Agent memory systems are explicitly identified in the framework cache as outside the framework's scope.

### RATC_16 - Semantic Memory and RAG
- Strength score: 1

RATC_16 describes RAG knowledge base poisoning where adversaries embed tool-invocation instructions in retrieved documents, knowledge graph relationship manipulation making unsafe tool combinations appear recommended, query rewriting instruction injection, and knowledge base staleness causing agents to invoke tools with obsolete formats. The framework explicitly identifies RAG pipelines and vector databases as gaps. SI-10 (input validation) and SI-7 (integrity checking) provide the most applicable controls—a knowledge base could be treated as a data store subject to integrity verification—but they do not address semantic poisoning where injected content is structurally valid but semantically malicious. SR-4 (provenance) could apply to tracking knowledge base data origins. CA-7 (continuous monitoring) could detect behavioral anomalies from stale or poisoned retrievals. These are weak indirect applications, and the core RAG-specific attack vectors are not addressed by the framework.

### RATC_17 - Utility Functions and Decision Logic
- Strength score: 1

RATC_17 describes exploitation of expected utility calculation in tool selection through outcome assumption injection, sequential utility miscalculation where intermediate agents misrepresent available future options, and trade-off weight manipulation in multi-objective utility functions. These are LLM and agentic decision architecture-specific vulnerabilities targeting the mathematical decision process by which agents select among tools. The framework has no concept of agent utility functions, multi-objective optimization in tool selection, or the manipulation of agent decision-making logic at this level of abstraction. SA-8 (security engineering principles) broadly encourages principled system design, and TEVV testing would evaluate model decision quality, but neither addresses runtime utility function poisoning or sequential multi-agent decision chain manipulation. This is an agentic-specific threat entirely outside the framework's scope.

### RATC_18 - Rule-Based and Knowledge-Engineered Systems
- Strength score: 1

RATC_18 covers attacks on rule-based tool authorization systems: specificity hierarchy exploitation where injected specific rules override safety rules, forward chaining exploitation through injected facts, certainty factor manipulation, and lexicographic heuristic objective reordering. While the framework addresses configuration management (CM family) for system rule and policy configurations—which would govern authorized rule sets—it does not address adversarial injection into rule-based reasoning engines at runtime or the exploitation of rule engine semantics (specificity ordering, certainty factors, forward chaining). CM-5 (access restrictions for change) and SI-7 could protect a rule base from unauthorized modification, but semantic injection attacks that work within the rule engine's own resolution mechanisms are outside the framework's threat model. These are knowledge-engineering-specific vulnerabilities with no CNSSI 1253 analog.

### RATC_19 - Learning and Reinforcement Learning
- Strength score: 1

RATC_19 describes RL-specific tool selection poisoning through reward signal manipulation, shared replay buffer poisoning in multi-agent RL systems, curriculum learning poisoning, imitation learning trajectory corruption, actor-critic cross-agent critic poisoning, and PPO trust region manipulation to iteratively converge agent policies toward attacker-controlled attractors. The framework's Phase 3 (AI Model Development) addresses model poisoning broadly and SI-7/SA-11 apply to integrity and testing of trained models. However, the framework cache notes that RL-specific training attacks beyond general poisoning are not addressed. Shared replay buffer poisoning, cross-agent critic value corruption, and trust region manipulation are RL-architecture-specific mechanisms with no CNSSI 1253 control mapping. The multi-agent RL coordination attack surface—where poisoning one agent's critic degrades all agents' policies—is an entirely novel threat category for the framework.

### RATC_21 - Parallel Retrieval Race Conditions in Multi-Agent Query Decomposition Systems
- Strength score: 2

RATC_21 describes race conditions in parallel multi-agent retrieval including shared index deduplication races producing inconsistent results, timeout cascade failures overwhelming shared connection pools causing fleet-wide outages, multi-stage caching temporal inconsistency from mismatched TTLs, and container registry supply chain compromise enabling fleet-wide backdoor deployment. The framework has two areas of direct applicability. SC-5 (DoS protection) and CP-2/4/9 (contingency planning and recovery) directly address the availability and resilience dimension of the connection pool exhaustion and fleet-wide outage scenarios. For the container registry supply chain attack, SR-3/4/6 and SI-7 (container integrity) provide strong applicable coverage. The retrieval race conditions and cache consistency failures are operational reliability issues where the framework's CA-7 (continuous monitoring) and IR-4 (incident handling) provide indirect applicability. The semantic consistency dimension of mixed-state query results is an agentic-specific concern not addressed, but the infrastructure-layer vectors have meaningful framework coverage.

### RATC_22 - Multi-Stage Pipeline Result Caching Race Enabling Multi-Agent Consistency Failures
- Strength score: 2

RATC_22 describes multi-stage cache TTL expiration creating temporal inconsistency windows, event-based invalidation propagation delays causing agents to serve data from different knowledge states, hash collision attacks enabling cache poisoning affecting all agents, warming inconsistency, container registry supply chain attacks (credential compromise, typosquatting, digest collision bypass, GitOps automation backdoors), tensor parallelism communication poisoning through multi-tenant memory isolation failures, and profiling telemetry exposure enabling targeted DoS and budget exhaustion. The framework covers several of these vectors. SR-3/4/6(1) and SI-7 apply strongly to the container registry supply chain attacks. SC-5 (DoS protection) and IR-4/5 address the DoS and budget exhaustion scenarios. SC-8/13 (cryptographic protection) apply to the tensor parallelism communication channel if treated as a network path. CM-6/7 (configuration management) govern cache and TTL configuration. The hash collision cache poisoning is closest to SI-10 (input validation) concerns. The semantic consistency and telemetry exposure vectors are less well-addressed, but the infrastructure and supply chain components receive meaningful framework coverage.

### RATC_27 - MIG Instance Co-Location Creating Shared Physical GPU Failure Propagation
- Strength score: 1

RATC_27 describes how MIG (Multi-Instance GPU) logical isolation operates atop shared physical GPU hardware—shared power regulators, thermal management, PCIe interfaces, GPU firmware, and driver stack—creating hidden dependencies where physical-layer failures or firmware vulnerabilities cause fleet-wide outages and potential cross-partition memory access breaking isolation. The framework explicitly identifies GPU-level hardware attacks as outside its scope. PE (physical and environmental) controls apply to data center physical access and environmental controls, which tangentially addresses power and thermal management at the facility level. SC-5 (DoS protection) and CP-9 (backup) address availability at the service level. SI-16 (memory protection) is conceptually relevant to the cross-partition memory access threat but is not designed for GPU VRAM isolation contexts. The framework's supply chain controls would apply to GPU firmware provenance (SR-4). However, MIG-specific co-location security, GPU driver stack vulnerabilities, and cross-partition isolation failures are hardware-level AI infrastructure concerns entirely outside the framework's ATLAS-derived threat model.

### RATC_29 - Shared Workflow State Version Conflict Amplification
- Strength score: 2

RATC_29 describes how optimistic locking in shared workflow state coordination creates systematic version conflicts under high concurrency, generating retry storms that intensify contention through a positive feedback loop, with targeted collision bursts exhausting retry budgets and causing workflow failures. This is a distributed systems reliability and availability threat with a clear DoS dimension. SC-5 (DoS protection) directly applies to the retry storm exhaustion scenario, which achieves denial of service through concurrency amplification rather than network flooding. CP-2/3/4 (contingency planning) and IR-4/8 (incident handling and response) address service degradation and recovery. CA-7 (continuous monitoring) and SI-4 (monitoring) would detect anomalous retry rates and version conflict spikes as behavioral deviations. The framework's Phase 4 threat model explicitly includes DoS as a threat vector with SC-5 as the control. The targeted collision burst technique for exhausting retry budgets maps to adversarial DoS patterns the framework recognizes. The multi-agent coordination-specific amplification dynamic is not addressed, but the framework's DoS and availability controls provide meaningful partial coverage.

### RATC_30 - Guardrail Bypass Through Infrastructure Failure Injection
- Strength score: 2

RATC_30 describes how guardrail validation infrastructure—GPU inference for safety classification, PII detection, and policy rule evaluation—operates under timeout thresholds, and adversary-induced processing delays or service unavailability (through network flooding, resource exhaustion, or DoS against centralized validation endpoints) cause safety validation to be skipped when agents fall back to delivering unvalidated responses. This threat has direct framework applicability across multiple control areas. SC-5 (DoS protection) directly addresses the network flooding and resource exhaustion attack vector against validation endpoints. SC-24 (fail in known state) is highly relevant: the framework explicitly requires systems to fail in a known, safe state rather than defaulting to unvalidated operation—a control directly applicable to the timeout-triggered bypass described here. CP-2/4 (contingency planning) and IR-4 (incident handling) address service unavailability and recovery. SI-6 (security function verification) requires verifying that security functions—including guardrails—are operating correctly, which encompasses detecting when validation services are unavailable. The combination of SC-5, SC-24, and SI-6 provides meaningful direct framework coverage for the infrastructure attack surface, though the LLM-specific guardrail and timeout bypass mechanism is not explicitly modeled.


## RDL — New data‑leakage channels via large contexts, logs, and probabilistic recall

### RDL_1 - UI/UX Patterns
- Strength score: 2

RDL_1 covers a cluster of UI/UX leakage channels unique to multi-agent chat interfaces: conversation history accumulating PII and credentials as a persistent repository, progressive disclosure debug views exposing inter-agent tokens and connection strings, session persistence bridging security boundaries, command palette history logging cross-agent behavioral sequences, dashboard aggregation correlating compartmentalized data sources, reference link metadata creating traversable information-flow graphs, approval workflow audit trails documenting distributed decision architecture, evidence links bypassing role-based access, iteration history exposing agent decision logic, and ARIA/alt-text inadvertently embedding sensitive values for scraping. The framework has meaningful indirect coverage. AU-2/3/9/12 (audit event logging and protection) are directly applicable to conversation history and approval trail logging—these controls require defining what events are logged and protecting log integrity. AC-3 and AC-6 (least privilege, access enforcement) apply to role-based access on evidence links and approval data. SI-12 and PT-2 through PT-4 (information management and PII handling) address the data minimization conflict that command palette history and session persistence create. AC-23 (data mining protection) partially covers the dashboard correlation leakage scenario. However, the framework has no guidance on UI/UX design, progressive disclosure security, ARIA attribute control, or the specific problem of multi-agent chat interfaces co-mingling data from agents operating under different access control regimes. The root design-level vulnerabilities are outside the ATLAS-derived threat model.

### RDL_2 - Data Persistence and Caching
- Strength score: 2

RDL_2 describes three persistence-layer leakage channels: session serialization converting unstructured conversation into queryable structured records (including rejected reasoning branches and inter-agent negotiation metadata), performance caching of multi-agent composite responses persisting sensitive aggregated data beyond session lifetime and permission revocations, and undo functionality maintaining recoverable copies of data users believe deleted—conflicting with data minimization regulations and creating distributed coordination complexity across agents. The framework has applicable controls. SC-28/28(1) (protection of information at rest with cryptographic protection) directly addresses session storage and cache confidentiality. SI-12/12(1)/12(2)/12(3) (information management and retention, limiting PII, minimizing PII in testing, information disposal) are directly relevant to the undo buffer persistence and right-to-erasure conflict. AC-4 (information flow enforcement) applies to cross-agent cache isolation. PT-3/PT-4 (PII processing and consent conditions) address the data minimization dimension. CA-7 (continuous monitoring) would detect anomalous access to session stores. The framework's gap is that it does not address the multi-agent cache key design problem—imperfect isolation between composite cache entries from agents with different security postures—or the distributed undo coordination problem that causes synchronized buffer clearing to fail across agent boundaries.

### RDL_3 - Streaming and Token-Level Leakage
- Strength score: 1

RDL_3 covers token-by-token streaming leakage where sensitive content appears in UI buffers and network streams before redaction can apply, tokenization boundary observation enabling context window inference attacks, output length correlation allowing session activity reconstruction from metadata alone, intermediate state exposure during multi-agent streaming cycles, streaming response caching creating cache side-channel exfiltration paths, and batched streaming revealing inter-agent processing assignments. These are LLM-specific and streaming-architecture-specific leakage channels that have no direct analog in traditional systems. SC-8/8(1) (transmission confidentiality and integrity) applies to encrypting streaming channels in transit, providing partial coverage for network-level interception. AU-3/12 (audit content and generation) would log streaming events after the fact. SI-15 (information output filtering) is the most conceptually applicable control—requiring filtering of information outputs—but the framework does not contemplate token-by-token progressive output where the filtering window lags behind generation. The streaming timing side-channel (output length correlation, tokenization observation) is entirely outside the framework's ATLAS-derived threat model, which does not include LLM inference-time observable side channels.

### RDL_4 - Search and Recall
- Strength score: 1

RDL_4 describes probabilistic recall through conversation search, where semantic similarity queries surface sensitive data from semantically related but policy-distinct agent domains—a vulnerability specific to multi-agent systems that index all agents' histories into a unified searchable corpus without security-boundary enforcement. AC-4 (information flow enforcement) and AC-6 (least privilege) are the most applicable controls, as they would require enforcing search boundaries consistent with data classification and role-based access. AC-23 (data mining protection) addresses protection against unauthorized data mining and aggregation, making it the most directly relevant control: an attacker systematically probing the unified search index with crafted semantic queries is a data mining attack. However, the framework's AC-23 implementation guidance does not address probabilistic semantic search, vector-space cross-contamination across agent domains, or the absence of access-controlled search boundaries in LLM conversation stores. The LLM-specific probabilistic recall mechanism—that semantic queries surface information the user did not explicitly request because the model treats the entire corpus as semantically unified—is not addressed by the framework.

### RDL_5 - Attribution and Observability
- Strength score: 2

RDL_5 covers attribution metadata logs exposing organizational structure and security priorities through agent usage distributions, framework-dependent logging aggregating multi-framework telemetry into comprehensive architecture disclosures, error classification patterns revealing system fragility maps, and latency measurement leakage enabling inference about computational load and tool availability. AU-2/6/7/13 (audit event definition, review, analysis, and monitoring for information disclosure) are directly applicable: AU-13 specifically addresses monitoring for unauthorized information disclosure in audit logs, and AU-6/7 require reviewing and analyzing audit logs for anomalies. SI-4 (system monitoring) and CA-7 (continuous monitoring) apply to detecting anomalous access to log aggregates. The framework's data confidentiality controls—SC-28 for logs at rest, SC-8 for transmission—protect log repositories. However, the specific threat that multi-framework log aggregation creates richer organizational intelligence than any single-framework log is an agentic architecture concern not addressed. The framework treats logging as a control mechanism, not as an attack surface itself; the concept of attribution logs as reconnaissance targets for competitive espionage is not modeled.

### RDL_7 - Tool Invocation and Function Calling
- Strength score: 2

RDL_7 describes a broad set of leakage channels arising from tool invocation logging and function-calling mechanics: comprehensive tool invocation logs leaking agent behavior and sensitive parameters; function calling JSON in context windows and conversation history persisting sensitive data across agent boundaries; tool error messages disclosing implementation details propagating through multi-agent conversation logs; tool selection reasoning stored as cognitive state leakage; function description enumeration revealing organizational capabilities; audit trail poisoning hiding malicious invocations; tool success rate differential leakage identifying high-performing agents for compromise; parameter values (API keys, PII, account numbers) leaking through centralized action log aggregation; accuracy metrics revealing operational patterns; and hybrid workflow log aggregation concentrating sensitive data. AU-2/3/10/12 (audit logging) are directly applicable to tool invocation logging and provide the framework's strongest coverage here—events and content captured in logs are within scope, and AU-10 (non-repudiation) addresses audit trail integrity relevant to the log poisoning scenario (RDL_7_6). AC-23 (data mining protection) applies to systematic parameter extraction from aggregated logs. SC-28 (protection of information at rest) and SC-4 (information in shared resources) address centralized log repository access. The gaps are the LLM-specific aspects: cognitive state leakage through stored reasoning, semantic function description enumeration, and success-rate differential analysis as an attack vector are not modeled.

### RDL_9 - Multi-Agent Memory and State
- Strength score: 1

RDL_9 covers reflection memory becoming unauthorized cross-agent conduits enabling covert data channels, ReAct reasoning trace persistence creating forensic attack graphs exposing agent topology and decision logic, distributed trace correlation ID timing analysis leaking workflow patterns across tenants, shared blackboard memory access pattern analysis enabling workflow reconstruction without content access, and dashboard correlation revealing business workflows through simultaneous agent activity patterns. AU-13 (monitoring for information disclosure) and SI-4 (system monitoring) are relevant to detecting anomalous cross-agent data access. AC-4 (information flow enforcement) and SC-4 (information in shared resources) apply to shared memory blackboard access control. CA-7 (continuous monitoring) would detect anomalous patterns. However, the framework does not address agent memory architectures, reflection memory, ReAct reasoning trace persistence, or distributed trace correlation as security concerns. The timing side-channel (cross-tenant correlation through trace metadata even when content is encrypted) is an LLM/agentic-specific observability attack not modeled by ATLAS-derived controls. The non-deterministic probabilistic recall that obscures leakage detection is entirely outside the framework's threat model.

### RDL_10 - Reasoning and CoT Traces
- Strength score: 1

RDL_10 covers chain-of-thought trace leakage through shared memory queries surfacing sensitive embedded context, intermediate step verbosity enabling input reconstruction from logged reasoning (e.g., inferring actual customer data from filter-step descriptions), reasoning-embedded secrets documented in traces shared across agents, cross-agent reasoning context replication as Agent A's full context flows to Agent B and gets logged, lookahead branch observation revealing future plans in Tree-of-Thought agents, preserved high-quality reasoning path leakage across agents, memory consolidation inadvertently preserving sensitive data in merged outputs, self-consistency intermediate chain probabilistic recall, quality score metadata access patterns as side-channel leakage, and failure case logs becoming training data for other agents. The framework explicitly identifies reasoning trace leakage as outside its scope. SI-12 (information management and retention) loosely applies to how long reasoning traces are retained. AU-9 (audit log protection) covers protecting trace logs from unauthorized access. However, no CNSSI 1253 control addresses CoT reasoning as an LLM mechanism, the security implications of explicit intermediate reasoning steps, cross-agent reasoning context propagation, or probabilistic recall through multi-agent memory retrieval. ### RDL_11 - Multimodal and Input Processing

### RIP_1 - Multi-Agent UI & Dashboard Attacks
- Strength score: 1

### RIP_2 - Approval & Workflow Attacks

The IA and AU families assume well-defined human principals with authenticated identities and do not model pipeline stages as distinct attribution subjects. Approval workflow integrity for agentic systems, especially HITL workflows spanning heterogeneous agent policies, is outside the framework's operational scope.
