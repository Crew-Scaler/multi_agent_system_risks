# Risk Management Analysis: DoD Responsible AI Strategy and Implementation Pathway

Framework: DoD Responsible AI (RAI) Strategy and Implementation Pathway (CDAO, October 2024 update of June 2022 document)

## Framework Overview

The DoD RAI Strategy is a governance and policy document organized around six Foundational Tenets: (1) RAI Governance, (2) Warfighter Trust, (3) AI Product and Acquisition Lifecycle, (4) Requirements Validation, (5) Responsible AI Ecosystem, and (6) AI Workforce. It operationalizes five DoD AI Ethical Principles: Responsible, Equitable, Traceable, Reliable, and Governable. The framework establishes Lines of Effort (LOEs) rather than technical controls. Security-relevant provisions include LOE 2.1.2d (tools to detect adversarial attacks on AI), LOE 2.1.6 (AI security research and adversarial defense), LOE 2.1.7 (DoD-wide AI security guidance covering supply chain and cybersecurity), the TEVV (Test, Evaluation, Verification, and Validation) framework requiring continuous lifecycle testing including for adversarial attacks, and the "Governable" principle requiring the ability to detect/avoid unintended consequences and disengage deployed systems with unintended behavior. As a governance framework, it primarily establishes what must be done rather than how, leaving specific technical controls to downstream documents such as NIST AI RMF and cybersecurity frameworks.

## RATC — Agent–tool coupling as "policy-level remote code execution"

### RATC_1 - UI/UX Abstraction and Visibility Gaps
- Strength score: 2

RATC_1 covers six threats where UI abstraction layers hide true tool chain complexity from human reviewers: approval UIs that mask SQL modifications behind benign descriptions, progressive disclosure concealing multi-agent execution graphs, inline suggestions executing tools invisibly, dashboards that fail to differentiate tools by risk level, context-driven tool selection without user awareness, and trace visualization that cannot display complete cross-agent tool chains. The DoD RAI Strategy's Warfighter Trust tenet directly addresses these concerns at the policy level. LOE 2.2.2 explicitly requires AI vendors to "provide traceable feedback on system status and clear procedures to trained operators to activate and deactivate system functions," and LOE 2.1.5 directs Human Systems Integration (HSI) guidance for system design, UX/UI, and user training. The "Traceable" AI Ethical Principle mandates transparent and auditable methodologies. The strategy does not, however, specify technical UI design requirements for multi-agent tool chain visibility, risk differentiation in dashboards, or cross-agent execution graph presentation — leaving implementation to program teams. The governance-level directives provide meaningful but indirect coverage.

### RATC_2 - Approval Workflow and User Attention Vulnerabilities
- Strength score: 2

RATC_2 covers approval fatigue enabling policy bypass, keyboard shortcut hijacking in approval workflows, and time-based default exploits that trigger unauthorized actions when reviewers are unavailable. The DoD RAI Strategy's Warfighter Trust tenet is directly relevant: LOE 2.2.2 requires traceable feedback and clear activation/deactivation procedures, LOE 2.2.1 mandates operator training on system capabilities, risks, and performance expectations, and LOE 2.1.2 directs development of T&E tools including adversarial attack detection. The "Governable" principle's requirement for the ability to disengage systems with unintended behavior applies to timeout-driven default-approve exploits. The strategy also directs (LOE 2.1.7) AI security guidance covering cybersecurity practices applicable to UI manipulation attacks. The strategy does not prescribe specific anti-fatigue mechanisms, multi-agent approval queue management, or UI-level security controls, so coverage is indirect and moderate.

### RATC_3 - Tool Overload, Parameter Handling, and Selection Errors
- Strength score: 1

RATC_3 covers tool overload privilege escalation across multi-agent architectures, tool argument injection enabling cross-boundary RCE, inline suggestion tool visibility gaps, and tool provenance obscuration in chat interfaces. These are specific technical attack patterns exploiting LLM-native weaknesses in parameter generation and validation. The DoD RAI Strategy's TEVV framework (LOE 2.1.1–2.1.4) requires testing AI capabilities for security and failure modes throughout the lifecycle, and LOE 2.1.2d directs tools to detect adversarial attacks on AI systems. The "Reliable" principle requires safety and security testing within defined uses. However, the strategy provides no specific guidance on parameter injection defenses, tool overload mitigations, argument validation across agent trust boundaries, or chat interface provenance requirements. These are highly technical agentic security concerns requiring specific engineering controls that the governance-level strategy does not address.

### RATC_4 - Confidence Manipulation and Tool Authorization
- Strength score: 2

RATC_4 covers adversarial prompt injection to inflate confidence scores and bypass approval gates, cascading failures when multi-agent systems trust confidence scores without re-validation, and tool authorization scope confusion enabling privilege escalation through compromised coordination agents. The DoD RAI Strategy's LOE 2.1.2d explicitly directs development of "tools and technologies required to detect both natural degradation of and adversarial attacks on AI" including notifying operators when attacks occur — directly applicable to confidence score manipulation as an adversarial attack. LOE 2.1.6 directs research into "AI security and defense to protect against adversarial attacks." The "Reliable" principle requires safety and effectiveness testing including against adversarial inputs. The "Governable" principle requires detecting and avoiding unintended consequences. The strategy does not specify confidence score validation architecture or multi-agent trust chain verification — those are implementation concerns — but the adversarial attack detection mandate and research direction provide moderate coverage.

### RATC_6 - Tool Metadata Poisoning Across Registries and Discovery
- Strength score: 2

RATC_6 covers centralized tool registry poisoning where compromising a shared registry manipulates how all agents understand and invoke tools, framework-specific tool selection poisoning, and framework abstraction leakage creating hidden tool chain exploitation. These are supply chain attacks on AI tool infrastructure. The DoD RAI Strategy explicitly identifies supply chain vulnerabilities as a risk in the "Desired End State" section, warning that "adversaries may seek to exploit supply chain vulnerabilities and inject themselves into critical AI training, testing, and update cycles." LOE 2.1.7 directs DoD-wide AI security guidance that "leverages existing best practices for risk management, supply chain security, and cybersecurity." LOE 3.1.3 addresses supply chain risk in acquisition by preserving government IP and avoiding proprietary lock-in. The strategy does not specify technical controls for tool registry integrity, cryptographic signing of tool metadata, or multi-agent registry isolation — limiting coverage to indirect supply chain risk management principles.

### RATC_8 - Advanced Tool Invocation Patterns and Inference
- Strength score: 1

RATC_8 covers tool chain exploitation through multi-agent replanning loops, tool result injection and observation manipulation in ReAct patterns, and tool schema variation attacks at cross-framework boundaries. These are sophisticated multi-agent exploitation patterns targeting specific architectural primitives. The DoD RAI Strategy's TEVV framework requires testing AI capabilities for failure modes and adversarial attacks (LOE 2.1.2), and LOE 2.1.6 directs research into AI security and adversarial defenses. However, these provisions establish goals without providing specific technical countermeasures. The specific attacks — progressively chaining tools through replanning failures, poisoning observation streams, exploiting schema translation layers — require architectural defenses and validation controls that the governance-level strategy cannot specify. The strategy's adversarial testing mandate provides weak structural support but no substantive coverage.

### RATC_9 - Web and Multimodal Tool Exploitation
- Strength score: 1

RATC_9 covers web agent vulnerabilities where adversaries inject instructions into web content through scraping tools, hidden form fields, and navigation logging manipulation, and multimodal retrieval vulnerabilities where poisoned visual/audio content causes agents to invoke malicious tools. These are adversarial content injection attacks against specific agent capabilities. LOE 2.1.2d directs tools to detect adversarial attacks on AI, and LOE 2.1.6 includes research into AI security against adversarial attacks. Both provisions nominally apply to instruction injection through web content and multimodal poisoning. However, the strategy provides no technical guidance on web scraping sandboxing, multimodal content validation, cross-modal parameter sanitization, or vision model output integrity. The governance requirement to detect and defend against adversarial attacks provides only weak structural coverage for these highly specific exploit patterns.

### RATC_10 - Efficiency Optimization and Resource Constraints
- Strength score: 1

RATC_10 covers token allocation attacks where upstream agents operating under tight token budgets generate incomplete tool descriptions inherited by downstream agents, and iteration budget and temperature tuning exploits that create policy inconsistency across agent types. These attacks exploit resource-constrained inference environments to degrade security properties. The DoD RAI Strategy touches on resource allocation under the implementation approach ("ensure the right balance between responsibility, speed, and ease of implementation"), but does not address agentic token/context budget exploitation. The TEVV framework could theoretically include testing for resource-constrained behavior, and the "Reliable" principle requires capabilities to perform as designed within their defined uses. Neither provision addresses the specific security implications of token budget constraints creating tool description truncation, and the strategy is entirely silent on temperature tuning attacks. Coverage is minimal.

### RATC_11 - Tool Execution Infrastructure and Orchestration
- Strength score: 1

RATC_11 covers API gateway routing policy manipulation redirecting tool calls to attacker-controlled services, and Kubernetes container security context modification enabling privilege escalation affecting all agent containers simultaneously. These are ICT infrastructure and container orchestration attacks. LOE 2.1.7 directs DoD-wide AI security guidance "leveraging existing best practices for risk management, supply chain security, and cybersecurity" — the cybersecurity component provides structural authority for container security and API gateway hardening. However, the DoD RAI Strategy does not itself provide such controls, delegating to existing cybersecurity frameworks. The strategy's role is to direct that security guidance be developed, not to provide it. Until downstream guidance materializes, the strategy offers only weak coverage through its directive mandate.

### RATC_12 - Distributed and Hardware-Level Attacks
- Strength score: 1

RATC_12 covers tensor parallelism communication interception where unencrypted inter-GPU communication can corrupt tool-calling policy decisions, and KV cache poisoning enabling cross-agent cache pollution and quantization calibration attacks. These are hardware-level and distributed inference attacks with no analog in the DoD RAI Strategy's governance scope. The strategy's TEVV framework and "Reliable" principle apply only at the level of requiring safety and security testing — they provide no guidance on GPU communication encryption, multi-tenant inference isolation, or quantization security. LOE 2.1.6 directs research into AI security topics that might eventually encompass hardware-level concerns, but as a governance document the DoD RAI Strategy has no applicable provisions for tensor parallelism or KV cache security.

### RATC_14 - Reasoning and Planning Vulnerabilities
- Strength score: 1

RATC_14 covers chain-of-thought reasoning quality failures enabling unsupported tool selections, poisoned reasoning traces embedding dangerous tool justifications, Self-Consistency voting manipulation through sampling path divergence, and hierarchical task network planning creating tool visibility fragmentation. These attack the cognitive architecture of LLM-based agents. The DoD RAI Strategy's "Traceable" principle requires auditable methodologies that could apply to CoT reasoning traces, and the TEVV framework requires testing for failure modes. LOE 2.1.2d requires tools to detect adversarial attacks — CoT trace poisoning is an adversarial attack. However, the strategy has no provisions for reasoning trace integrity, voting manipulation defenses, or hierarchical planning transparency. The governance-level adversarial attack detection directive is too abstract to provide substantive coverage for these specific reasoning architecture vulnerabilities.

### RATC_15 - Episodic Memory and Learning
- Strength score: 1

RATC_15 covers episodic memory poisoning where attackers store malicious tool sequences as "successful resolutions" causing agents to replicate dangerous chains, with multi-agent shared episodic memory enabling fleet-wide propagation and trajectory abstraction converting poisoned episodes into procedural workflows. These are memory-system attacks with no direct coverage in the DoD RAI Strategy. The supply chain guidance (LOE 2.1.7) applies at the broadest level since poisoned episodic memory is a form of data poisoning, and TEVV requirements for lifecycle testing could include memory integrity testing. However, the strategy provides no guidance on episodic memory validation, episode provenance verification, or isolation of episodic memory across agents. Coverage is minimal and only structural.

### RATC_16 - Semantic Memory and RAG
- Strength score: 1

RATC_16 covers RAG pipeline and knowledge base poisoning where adversaries embed tool-invocation instructions in retrieved content, knowledge graph relationship poisoning causing unsafe tool combinations to appear recommended, and knowledge base staleness causing agents to invoke tools with obsolete formats. The DoD RAI Strategy mentions protecting AI training, testing, and update cycles from supply chain exploitation (Desired End State), and LOE 3.2.3 directs AI Data Cards to document data assets. LOE 2.1.7 covers AI security including data security. However, none of these provisions address RAG-specific attack vectors, embedding space manipulation, or knowledge graph integrity. The data documentation requirements (Data Cards) could improve awareness of knowledge base provenance but do not constitute technical defenses. Coverage is weak.

### RATC_17 - Utility Functions and Decision Logic
- Strength score: 1

RATC_17 covers expected utility calculation vulnerabilities through outcome assumption injection causing agents to miscalculate tool utility, sequential utility miscalculation, and trade-off weight manipulation poisoning multi-objective utility functions. These are attacks on AI decision logic foundations. The DoD RAI Strategy has no applicable provisions for utility function security. The "Equitable" principle addresses bias minimization which could tangentially apply to manipulated utility weights, and the TEVV framework requires testing AI capabilities. Neither provides substantive coverage for utility function poisoning, which requires specific architectural defenses and formal verification approaches not contemplated in the governance strategy.

### RATC_18 - Rule-Based and Knowledge-Engineered Systems
- Strength score: 1

RATC_18 covers rule-based authorization bypass through specificity hierarchy attacks enabling injected rules to override safety rules, forward chaining exploitation, certainty factor manipulation, and lexicographic objective reordering. These are attacks on symbolic AI components. The DoD RAI Strategy's TEVV framework provides weak structural coverage — testing should theoretically catch rule-based authorization bypass. The "Reliable" principle requires capabilities to function safely and effectively. However, the strategy provides no guidance on rule engine security, specificity hierarchy integrity, or certainty factor validation. These highly specific technical attacks on rule-based reasoning systems are entirely outside the governance document's scope.

### RATC_19 - Learning and Reinforcement Learning
- Strength score: 1

RATC_19 covers RL tool selection policy hijacking through reward poisoning, shared replay buffer poisoning affecting all agents' Q-values, curriculum learning poisoning, imitation learning trajectory poisoning, actor-critic cross-agent critic poisoning, and proximal policy optimization trust region manipulation. These are ML training-time attacks. The DoD RAI Strategy explicitly warns about supply chain attacks on "critical AI training, testing, and update cycles" (Desired End State) and LOE 2.1.7 covers supply chain security for AI — providing structural relevance to training-time attacks. However, the strategy does not address specific RL attack vectors, shared replay buffer security, or gradient-based policy manipulation defenses. The supply chain warning is generic and the strategy offers only minimal structural coverage for these training-time RL attacks.

### RATC_21 - Parallel Retrieval Race Conditions in Multi-Agent Query Decomposition Systems
- Strength score: 1

RATC_21 covers parallel retrieval race conditions creating database inconsistency, shared index deduplication races, timeout-triggered connection pool exhaustion causing fleet-wide outages, multi-stage caching temporal inconsistency, and container registry supply chain vulnerabilities enabling fleet-wide poisoning. These are distributed systems reliability and supply chain security issues. LOE 2.1.7 covers supply chain security applicable to container registry attacks, and the "Reliable" principle requires safety and effectiveness testing. The TEVV framework could include availability and race condition testing. However, parallel retrieval race conditions, cache TTL inconsistency, and fleet-wide connection pool exhaustion are distributed systems engineering problems requiring specific infrastructure controls not provided by the governance-level DoD RAI Strategy.

### RATC_22 - Multi-Stage Pipeline Result Caching Race Enabling Multi-Agent Consistency Failures
- Strength score: 1

RATC_22 covers multi-stage pipeline caching races creating temporal windows of inconsistent results, event-based cache invalidation propagation delays causing divergent fleet state, shared cache write races, hash collision attacks poisoning cache entries affecting all agents, and supply chain attacks through container registry credential compromise. The supply chain component — registry credential compromise, typosquatting, digest collision bypass, GitOps automation backdoors — falls under the DoD RAI Strategy's supply chain security concern (Desired End State, LOE 2.1.7). Cache race conditions and consistency failures are infrastructure engineering issues outside the strategy's scope. The supply chain relevance provides minimal structural coverage for the container registry attack vectors, but cache timing attacks and write races receive no coverage.

### RATC_27 - MIG Instance Co-Location Creating Shared Physical GPU Failure Propagation
- Strength score: 1

RATC_27 covers MIG (Multi-Instance GPU) logical isolation failures where shared physical GPU components — power regulators, thermal management, PCIe interface, firmware, driver stack — create hidden fleet-wide failure propagation despite partition isolation, including firmware vulnerabilities enabling cross-partition memory access. These are hardware-level GPU infrastructure attacks. The DoD RAI Strategy has no applicable provisions for hardware isolation security, GPU firmware integrity, or multi-tenant compute security. The "Reliable" principle and TEVV framework apply only at a very high level of requiring AI systems to work as designed. This specific hardware attack surface is entirely outside the governance document's scope.

### RATC_29 - Shared Workflow State Version Conflict Amplification Through Concurrent Write Flooding
- Strength score: 1

RATC_29 covers optimistic locking version conflicts in shared workflow state where high agent concurrency creates retry storms that intensify rather than resolve contention, with conflict probability scaling quadratically and targeted collision bursts exhausting retry budgets to cause workflow failures. These are distributed systems concurrency attacks causing denial-of-service against agentic workflows. The DoD RAI Strategy's "Reliable" principle requires AI capabilities to function safely and effectively, and TEVV testing could theoretically include concurrency and load testing. However, the strategy provides no guidance on distributed state management, retry storm mitigation, or optimistic locking security. These are pure distributed systems engineering problems outside the governance framework's scope.

### RATC_30 - Guardrail Bypass Through Infrastructure Failure Injection Preventing Safety Validation Execution
- Strength score: 2

RATC_30 covers adversary-induced processing delays exceeding guardrail validation timeouts causing responses to bypass safety checks, and service unavailability attacks (DoS against centralized validation endpoints) causing fleet-wide guardrail bypass when agents fall back to unvalidated responses. This is a specific attack against AI safety infrastructure availability. The DoD RAI Strategy directly addresses this class of concern through the "Governable" principle — AI systems must "possess the ability to detect and avoid unintended consequences, and the ability to disengage or deactivate deployed systems that demonstrate unintended behavior." Infrastructure failures causing safety validation bypass represent exactly the unintended behavior the Governable principle is designed to prevent. LOE 2.1.2 requires tools to detect adversarial attacks, and DoS against safety validation is an adversarial attack. LOE 2.1.7 directs cybersecurity guidance covering DoS protection. The strategy's high-level safety assurance requirements and adversarial attack detection mandate provide moderate indirect coverage for guardrail bypass through infrastructure failure.


## RDL — New data-leakage channels via large contexts, logs, and probabilistic recall

### RDL_1 - UI/UX Patterns
- Strength score: 2

RDL_1 covers eleven threats arising from UI design choices: chat histories accumulating multi-domain sensitive data into unified scrollable repositories, progressive disclosure debug views exposing inter-agent authentication tokens and connection strings, session persistence storing aggregated context across security-level transitions, command palette histories logging cross-agent operation sequences, multi-agent dashboards revealing cross-domain data correlations, context reference links disclosing relational metadata through link structure, approval workflow audit trails documenting complete distributed decision architectures, evidence links creating indirect access paths bypassing role-based permissions, interactive refinement histories exposing agent decision logic and quality thresholds, and alt text/ARIA labels embedding sensitive data in accessibility structures. The DoD RAI Strategy's "Traceable" AI Ethical Principle mandates transparent and auditable methodologies, implicitly requiring that audit logging be secure and access-controlled. LOE 3.2.5 directs publishing best practices "to preserve privacy and civil liberties and to avoid unintended bias in the design and development of AI capabilities that involve the use of personal information" — applicable to UI-mediated PII leakage. LOE 2.2.2 requires traceable feedback mechanisms. The strategy does not address specific UI security design patterns, multi-agent session security boundaries, or cross-domain data aggregation controls. Coverage is indirect through privacy and traceability principles.

### RDL_2 - Data Persistence and Caching
- Strength score: 1

RDL_2 covers session persistence serialization converting conversations into queryable structured records for efficient exfiltration, cached responses persisting sensitive multi-agent composite data beyond session lifetime with weaker access controls than primary systems, and undo functionality maintaining recoverable copies of deleted data across distributed agents that conflict with data minimization obligations. LOE 3.2.5 directs privacy and data minimization best practices — providing structural relevance to cache and session retention security. The "Responsible" principle includes appropriate care in data handling. However, the DoD RAI Strategy provides no guidance on session serialization security, cache access control architecture, multi-agent distributed undo buffer management, or retention policy enforcement across agent systems. These require specific technical and engineering controls the governance document does not specify.

### RDL_3 - Streaming and Token-Level Leakage
- Strength score: 1

RDL_3 covers streaming response token-by-token leakage before post-processing redaction can apply (with multi-agent streaming handoffs creating multiple sequential exposure windows), tokenization pattern observation enabling context window inference attacks across heterogeneous agents, streaming output length correlation enabling activity pattern reconstruction from metadata, streaming state updates leaking progressive intermediate states in multi-agent cycling workflows, intermediate streaming state disclosure through batch processing, and streaming response caching creating N-party visibility into sensitive streamed content. These are highly specific technical data leakage channels unique to LLM streaming inference. The DoD RAI Strategy does not address streaming security, token-level exposure windows, tokenization side-channels, or streaming cache access controls. The "Traceable" and "Reliable" principles apply only at a very abstract level. No substantive coverage exists.

### RDL_4 - Search and Recall
- Strength score: 1

RDL_4 covers probabilistic recall through conversation search where semantic queries surface sensitive data from cross-domain agent histories without enforcing access policy boundaries — the unified multi-agent corpus enables systematic probing with crafted queries designed to surface high-sensitivity content through semantic association. The DoD RAI Strategy's LOE 3.2.5 privacy guidance and "Traceable" principle apply at the broadest level. The specific problem — that semantic search treats entire multi-agent history as a unified corpus ignoring organizational access boundaries — requires technical controls such as access-controlled semantic search, per-domain index isolation, or query-time authorization filtering that the governance strategy does not address.

### RDL_5 - Attribution and Observability
- Strength score: 1

RDL_5 covers attribution metadata logs exposing organizational structure and operational workflows (agent usage distributions mapping hierarchy and high-privilege users), framework-dependent logging aggregating implementation details from multiple frameworks into comprehensive architecture disclosures, error classification pattern leakage revealing infrastructure fragility mapping, and latency measurement leakage enabling attack timing optimization. The "Traceable" principle requires auditable logging, creating a tension where required audit logs also become high-value intelligence targets. The DoD RAI Strategy does not address log access controls, telemetry data classification, or limiting the operational intelligence extractable from compliance-required audit records. The need for traceability mandated by the strategy may actually increase the attack surface for these metadata-based leakage channels.

### RDL_7 - Tool Invocation and Function Calling
- Strength score: 2

RDL_7 covers ten threats: tool invocation logs as side channels disclosing agent behavior and data flows, function calling JSON persisting sensitive parameters in context windows and conversation histories, tool error messages leaking implementation details enabling exploitation, tool selection reasoning as cognitive state leakage profiling decision-making, function description enumeration exposing organizational capabilities, audit trail poisoning where compromised agents omit malicious invocations, tool success rate differential leakage identifying high-performing agents to compromise, parameter values leaking PII and credentials through centralized log repositories, action accuracy metrics revealing operational patterns, and log aggregation concentrating sensitive data from all agents. The DoD RAI Strategy's "Traceable" principle requires auditable methodologies and data sources — specifically applicable to tool invocation logging requirements. LOE 2.2.3 directs monitoring performance with guidance on instrumentation tools. The audit trail poisoning threat (RDL_7_6) is directly addressed by the "Responsible" principle requiring accountability and by governance requirements for reliable audit infrastructure. LOE 2.1.7 covers cybersecurity guidance applicable to log access controls. While these provisions establish the requirement for secure audit infrastructure, the specific technical controls for parameter sanitization, log access management, and cross-agent audit integrity are implementation concerns left to downstream guidance.

### RDL_9 - Multi-Agent Memory and State

RDL_9 covers cross-agent reflection memory leakage creating unauthorized conduits across trust boundaries, ReAct reasoning trace persistence exposing complete attack graphs with agent topology and tool dependencies, distributed trace correlation leakage enabling workflow pattern inference through timing metadata in multi-tenant environments, shared memory blackboard access pattern analysis enabling workflow reconstruction without content access, and multi-agent dashboard correlation leakage through simultaneous agent activity patterns. ## RIP — Agent identity, provenance, and economic/resource attack surface

### RIP_1 - Multi-Agent UI & Dashboard Attacks
- Strength score: 2

RIP_1 covers four attacks where multi-agent dashboard UIs lack clear identity indicators: dashboard identity spoofing through attribution gaps enabling impersonation of high-authority agents, forensics failures preventing post-incident attribution across multi-agent pipelines, real-time cost attribution failures creating economic accountability gaps, and GroupChat speaker attribution manipulation enabling social engineering through false identity claims. The DoD RAI Strategy's "Traceable" AI Ethical Principle explicitly requires auditable methodologies — directly applicable to agent identity attribution and dashboard forensics requirements. LOE 2.2.2 requires AI vendors to "provide traceable feedback on system status and clear procedures to trained operators to activate and deactivate system functions," addressing the dashboard transparency dimension of these attacks. LOE 2.1.5 directs Human Systems Integration guidance for system design and UX/UI relevant to multi-agent dashboard requirements. The strategy does not specify technical mechanisms for cryptographic agent identity binding, per-agent cost attribution in dashboards, or cross-agent forensics tooling, so coverage is indirect and governance-level.
