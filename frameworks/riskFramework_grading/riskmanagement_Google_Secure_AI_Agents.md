# Risk Management Analysis: Google's Approach for Secure AI Agents

**Framework:** Google's Approach for Secure AI Agents: An Introduction (Santiago Díaz, Christoph Kern, Kara Olive)
**Source:** `Google_Secure_AI_Agents/Google-Approach-Secure-AI-Agents.pdf`

## Framework Summary

Google's Secure AI Agents framework addresses two primary agent risk categories — rogue actions (unintended, harmful, or policy-violating behaviors) and sensitive data disclosure — through three core security principles and a hybrid defense-in-depth strategy.

The three principles are: (1) **Agents must have well-defined human controllers** — every agent must have attributable controlling human users, with explicit confirmation required for critical or irreversible actions; (2) **Agent powers must have limitations** — dynamic, purpose-aligned least privilege enforced via robust Authentication, Authorization, and Auditing (AAA) infrastructure, scoped OAuth tokens, and sandboxing; (3) **Agent actions and planning must be observable** — robust logging of inputs, tools invoked, parameters, outputs, and reasoning steps, with transparent UIs.

The hybrid defense-in-depth strategy combines: **Layer 1** (traditional deterministic policy engines operating outside the AI model's reasoning, intercepting action requests and evaluating them against predefined rules, with outcomes of allow/block/require confirmation) and **Layer 2** (reasoning-based defenses including adversarial training, specialized guard model classifiers, and plan analysis/prediction models). Continuous assurance activities (regression testing, variant analysis, red teams) support both layers.

Key limitation: the framework is an application/architecture-level introductory paper. It does not address inference hardware security, GPU-level attacks, multi-agent RL training vulnerabilities, or the specific mechanics of distributed compute race conditions.

## Scoring Key

- **3** — Framework directly and strongly supports management of the risks (specific applicable guidance).
- **2** — Framework indirectly and moderately supports management (relevant principles or controls that partially apply).
- **1** — Framework weakly supports management (only general principles with minimal direct applicability).

## RATC — Agent–tool coupling as "policy-level remote code execution"

### RATC_1 - UI/UX Abstraction and Visibility Gaps
- Strength score: 2

Google's Principle 3 (Observability) directly targets the class of risks in this subsubsection: it mandates that "user interfaces should be designed to promote transparency, providing users with insights into the agent's thought process, the data sources it consulted, or the actions it intends to take, especially for complex or high-risk operations." The framework further requires that action properties (read-only vs. state-changing, handling of sensitive data) be clearly characterized as metadata for both automated mechanisms and human reviewers. Principle 1 also explicitly asks whether users can "inspect, approve, and revoke permissions for agent actions." These directives address the general failure mode of UIs abstracting away true tool invocation risk. However, the framework does not operationalize specific UI design patterns for multi-agent dashboard risk differentiation, progressive disclosure attack surface hardening, trace visualization complexity management across agent boundaries, or context-driven tool selection transparency in multi-agent pipelines. The guidance is directionally correct but lacks specificity for the multi-agent scenarios described in the threats.

### RATC_2 - Approval Workflow and User Attention Vulnerabilities
- Strength score: 1

Google's Principle 1 requires explicit human confirmation before proceeding with critical or irreversible actions, and Layer 1 policy engines can mandate user confirmation as an enforcement outcome. These provide conceptual relevance to the approval workflow category. However, the framework does not address approval fatigue as a systemic vulnerability in multi-agent systems that generate exponentially more approval requests, keyboard shortcut hijacking in approval UIs via injected JavaScript or focus manipulation, or time-based default exploitation where high-risk requests are timed during coverage gaps or reviewer overload. The human control principle focuses on ensuring confirmation exists, not on protecting the confirmation mechanism itself against exploitation.

### RATC_3 - Tool Overload, Parameter Handling, and Selection Errors
- Strength score: 2

Google's Principle 2 (Limited Powers) partially addresses tool overload privilege escalation through dynamic, context-aware least privilege: agents are to have "only the capabilities and permissions necessary for their intended purpose," with scoped OAuth tokens and AAA infrastructure. This constrains the privilege escalation impact of tool overload degradation and hierarchical delegation abuse. Principle 3's action metadata characterization and transparent UIs partially address tool provenance obscuration in chat and inline suggestion interfaces. The framework also notes that "dynamically incorporating new tools, especially third-party ones, introduces risks related to deceptive tool descriptions or insecure implementations." However, argument injection across multi-agent trust boundaries, parameter extraction errors compounding through chained calls, and the specific mechanics of LLM probabilistic generation enabling payload injection as parameter values are not addressed.

### RATC_4 - Confidence Manipulation and Tool Authorization
- Strength score: 2

Layer 1 policy engines are a key defense here: they operate deterministically "outside the AI model's reasoning process," intercepting and evaluating action requests against predefined rules before execution. Because the policy engine does not rely on the AI model's confidence scores to make allow/block decisions, confidence score inflation attacks targeting AI-internal authorization gates are substantially mitigated for actions subjected to policy enforcement. The framework's AAA infrastructure requirement and scoped credentials also partially address authorization scope confusion across agent specializations. However, the framework does not specifically address multi-agent confidence score propagation (where confidence scores from one agent are trusted by another without re-validation), precondition validation bypass through transitive trust across agent delegation chains, or confidence threshold manipulation in majority-voting-based multi-agent self-consistency systems.

### RATC_6 - Tool Metadata Poisoning Across Registries and Discovery
- Strength score: 2

The Google framework explicitly identifies that "dynamically incorporating new tools, especially third-party ones, introduces risks related to deceptive tool descriptions or insecure implementations," showing direct recognition of the tool metadata poisoning threat class. Principle 2's authentication and authorization requirements for tool use, scoped credentials, and AAA infrastructure provide relevant access control to limit who can modify tool registries. Layer 2 guard models (classifiers) could potentially detect malicious instruction patterns embedded in tool descriptions. However, the framework provides no specific controls for centralized tool registry integrity, vector database metadata poisoning via semantic search manipulation, few-shot parameter examples used as instruction injection vectors in tool descriptions, or framework-specific tool selection poisoning through decision context manipulation.

### RATC_8 - Advanced Tool Invocation Patterns and Inference
- Strength score: 2

The framework's prompt injection defenses are directly relevant to tool result injection and observation manipulation: structural prompt conventions (clear delimiters, role tagging) and Layer 2 guard models are proposed specifically to "differentiate trusted commands from untrusted inputs," which applies when one agent's tool output becomes another agent's observation. Layer 1 policy engines can intercept replanning-triggered action requests. The framework explicitly identifies iterative planning in a "reasoning loop" as a high-risk pattern requiring robust controls. However, the multi-agent-specific dynamics — cross-agent state accumulation without global consistency checks, streaming observation injection enabling progressive escalation across orchestration layers, and tool schema variation at cross-framework translation boundaries — are not specifically addressed.

### RATC_9 - Web and Multimodal Tool Exploitation
- Strength score: 2

The framework explicitly identifies prompt injection via "files, emails, or websites" as a primary threat mechanism and proposes structural defenses (delimiters, role tagging to separate untrusted external content from trusted instructions) and Layer 2 guard classifiers to detect instruction-like language in data. This directly addresses web agent prompt injection via adversarial web content and form hidden field injection. Input parsing and stream separation apply to web content handling. However, multimodal-specific attacks — vision model output laundering through sequential multi-agent processing causing malicious outputs to gain legitimacy, cross-modal parameter injection without sanitization boundaries, and multi-agent multimodal RAG poisoning — are not specifically addressed by the framework.

### RATC_10 - Efficiency Optimization and Resource Constraints
- Strength score: 1

The Google framework does not address LLM-specific inference parameters: token allocation, context window constraints, iteration budgets, temperature settings, or model selection diversity. Principle 3's observability requirement could theoretically surface resource usage patterns in logs, but no specific controls are provided for attacker-induced context constraints that truncate safety-critical tool descriptions in upstream agents, iteration budget exploitation creating policy inconsistency across the multi-agent fleet, or temperature tuning that separates actual accuracy from confidence visibility. These are inference-time efficiency parameter attacks outside the framework's scope.

### RATC_11 - Tool Execution Infrastructure and Orchestration
- Strength score: 2

The framework explicitly asks "What safeguards (such as sandboxing policies, user controls, and sensitive deployment exclusions) prevent agent actions from improperly exposing high-privilege information or capabilities in low-privilege contexts?" Principle 2's controls include sandboxing and robust AAA infrastructure. Authentication and authorization requirements for tool use apply to API gateway security. Policy engines create security chokepoints before action execution that can enforce routing constraints. However, specific container orchestration vulnerabilities — Kubernetes security context modifications, Persistent Volume Claim compromise, sidecar proxy interception of tool calls, init container configuration injection — and API gateway routing policy manipulation to redirect tool calls to attacker-controlled services are not specifically addressed within the framework.

### RATC_12 - Distributed and Hardware-Level Attacks
- Strength score: 1

The Google framework operates entirely at the application and architecture level and provides no guidance on GPU-level distributed compute attacks. Tensor parallelism communication interception (unencrypted inter-GPU traffic corrupting tool-calling policy decisions), KV cache poisoning via multi-tenant shared GPU inference biasing tool selection, and quantization calibration dataset poisoning causing downstream agents to receive systematically corrupted parameter outputs are AI inference hardware vulnerabilities entirely outside the framework's scope. No applicable controls exist.

### RATC_14 - Reasoning and Planning Vulnerabilities
- Strength score: 2

Google's Layer 2 reasoning-based defenses include "analyzing the agent's proposed plan of action" with models that "predict the probability of that plan leading to an undesirable outcome, potentially flagging high-risk plans for review or triggering stricter policy enforcement." This plan analysis capability is directly relevant to detecting dangerous tool sequences embedded in chain-of-thought reasoning and self-consistency voting manipulation. Adversarial training also exposes models to attack patterns. The framework explicitly identifies "iterative planning (in a reasoning loop)" as high-risk requiring robust controls. However, chain-of-thought trace poisoning propagating through shared reasoning memory to other agents, quality-weighted voting exploitation in RASC manipulating assessment metrics, and hierarchical task network planning fragmentation creating tool-awareness gaps between planning and execution layers are not specifically addressed.

### RATC_15 - Episodic Memory and Learning
- Strength score: 2

The Google framework directly addresses memory as a persistent attack vector: "If malicious data containing a prompt injection is processed and stored in memory, it could influence the agent's behavior in future, unrelated interactions." Memory implementations "must ensure strict isolation between users and potentially between different contexts for the same user to prevent contamination," and users need "transparency and control over agent memory." These controls apply to episodic memory stores where poisoned action trajectories could influence future tool-invocation behavior. Layer 2 guard models could potentially detect malicious patterns in retrieved episodes. However, the specific mechanism of cross-agent episodic propagation — where one compromised agent's poisoned episodes become learned policy for all agents sharing the same episodic memory store — and trajectory abstraction converting poisoned episodes into procedural workflows are not specifically addressed.

### RATC_16 - Semantic Memory and RAG
- Strength score: 2

The framework's memory security guidance (preventing persistent injection, memory isolation) and prompt injection defenses (structural separation of retrieved content from trusted instructions using delimiters and role tagging) apply to RAG pipeline attacks where knowledge base poisoning embeds tool-invocation instructions in retrieved content. Layer 2 guard models could detect malicious instruction patterns in RAG-retrieved documents. However, knowledge graph relationship poisoning causing unsafe tool combinations to appear recommended, query rewriting instruction injection affecting all agents' retrieval transformations, knowledge base staleness causing agents to invoke tools with obsolete formats, and adversarial retrieval triggers are not specifically addressed.

### RATC_17 - Utility Functions and Decision Logic
- Strength score: 1

The Google framework does not address utility functions, decision-theoretic planning, or reward/objective function integrity in AI agents. Outcome assumption injection causing agents to miscalculate tool utility, sequential expected utility miscalculation through misrepresented intermediate options, and trade-off weight manipulation in multi-objective utility functions are decision-logic layer attacks entirely outside the framework's guidance. Principle 2's dynamic privilege aligned with "agent's specific purpose and current user intent" is conceptually related to correct decision logic but provides no operational controls against utility function manipulation.

### RATC_18 - Rule-Based and Knowledge-Engineered Systems
- Strength score: 1

The framework's Layer 1 policy engines are themselves rule-based systems, and their external deterministic nature provides implicit integrity protection. AAA infrastructure controls who can modify system instructions and policies. However, specific rule-based AI authorization bypass patterns — specificity hierarchy exploitation to inject more-specific rules overriding safety constraints, forward chaining exploitation through fact injection triggering malicious authorization chains, certainty factor manipulation bypassing risk assessment, and lexicographic heuristic objective reordering — are knowledge-engineering-level attacks on inference engines not addressed by the framework.

### RATC_19 - Learning and Reinforcement Learning
- Strength score: 1

Google's Layer 2 adversarial training exposes models to examples of attacks and teaches safe responses, providing marginal relevance to RL-based attack categories. Assurance activities including red teams and regression testing could theoretically probe for RL-based vulnerabilities. However, the framework focuses on inference-time agent security rather than training-time RL manipulation. Reward poisoning hijacking learned tool selection policies, multi-agent RL shared replay buffer attacks affecting all agents' learned Q-values, curriculum learning poisoning, imitation learning trajectory corruption, actor-critic cross-agent critic poisoning, and proximal policy optimization trust region manipulation are RL training and multi-agent coordination vulnerabilities not addressed by the framework.

### RATC_21 - Parallel Retrieval Race Conditions in Multi-Agent Query Decomposition Systems
- Strength score: 1

The Google framework provides no specific guidance for parallel retrieval race conditions in multi-agent systems. Concurrent sub-query retrieval from inconsistent database states, shared index deduplication races, timeout cascade failures overwhelming connection pools, multi-stage caching temporal inconsistency from per-stage TTL expiry, and multi-agent cache consistency failures from network propagation delays are distributed systems coordination vulnerabilities outside the framework's scope. Container supply chain vulnerabilities (registry credential compromise) are nominally within the AAA infrastructure concept but are not operationalized. Observability logging could surface anomalous retrieval patterns but provides no mitigation.

### RATC_22 - Multi-Stage Pipeline Result Caching Race Enabling Multi-Agent Consistency Failures
- Strength score: 1

Similar to RATC_21, the Google framework provides no specific controls for multi-stage pipeline caching race conditions. Event-based cache invalidation propagation causing agents to serve data from different knowledge states, shared caching write races, hash collision attacks enabling adversaries to poison cache entries affecting all agents, and cache warming inconsistency across the fleet are distributed caching vulnerabilities outside the framework's guidance. Tensor parallelism inter-GPU communication poisoning via multi-tenant collocation and profiling telemetry exposure enabling targeted DoS are entirely outside scope.

### RATC_27 - MIG Instance Co-Location Creating Shared Physical GPU Failure Propagation Across Multi-Agent Deployments
- Strength score: 1

The Google framework operates at the application and architecture layer and provides no guidance on GPU hardware infrastructure security. MIG partitioning security, shared physical GPU hardware component failure propagation (power regulators, thermal management, PCIe interface, firmware), firmware vulnerabilities enabling cross-partition memory access, and CUDA driver bugs causing fleet-wide outages affecting all co-located MIG instances are infrastructure-level concerns entirely outside the framework's scope.

### RATC_29 - Shared Workflow State Version Conflict Amplification Through Concurrent Write Flooding Creating Retry Storm Cascades
- Strength score: 1

The framework's observability principle requires logging of agent actions and inputs, which could theoretically surface unusual retry storm patterns for human review. However, the framework provides no specific controls for distributed workflow state management, optimistic locking protocol security, concurrent write flooding inducing version conflict amplification, or the positive feedback loop where retry storms intensify contention quadratically with agent concurrency. These are multi-agent distributed coordination exploitation patterns not addressed by any of the framework's three principles or its hybrid defense layers.

### RATC_30 - Guardrail Bypass Through Infrastructure Failure Injection Preventing Safety Validation Execution
- Strength score: 1

The Google framework establishes policy engines and reasoning-based guard models as the guardrail system and notes that the hybrid defense aims to "constrain potential harm" even when agent reasoning is compromised. However, the framework does not address the security of the guardrail infrastructure itself. Adversary-induced processing delays exceeding validation timeout thresholds, network flooding or resource exhaustion causing fleet-wide guardrail bypass when agents fall back to delivering unvalidated responses, and the fundamental fail-open vs. fail-closed design question for validation infrastructure under DoS attack are not addressed. The framework notes that reasoning-based defenses are "non-deterministic and cannot provide absolute guarantees," but does not provide guidance on infrastructure resilience for the deterministic policy engines either.

## RDL — New data‑leakage channels via large contexts, logs, and probabilistic recall

### RDL_1 - UI/UX Patterns
- Strength score: 2

The Google framework's Principle 3 (Observability) and its emphasis on transparent UIs provide directional coverage for this subsubsection. The framework requires that UIs give users visibility into agent reasoning, data sources, and intended actions, and explicitly asks whether users can inspect and revoke agent permissions. The framework also directly identifies sensitive data disclosure as a primary risk category and requires that outputs be sanitized (XSS prevention, proper sanitization) before being rendered. These controls are relevant to the chat interface accumulation of credentials and PII (RDL_1_1), debug views exposing tokens and connection strings (RDL_1_2), and approval workflow audit trails aggregating inter-agent reasoning (RDL_1_8). However, the framework provides no specific guidance for multi-agent dashboard cross-domain correlation leakage (RDL_1_5), session persistence across security boundary transitions (RDL_1_3), accessibility attribute exploitation via ARIA labels (RDL_1_11), or command palette history as behavioral reconnaissance (RDL_1_4). The framework's UI transparency guidance is oriented toward ensuring users see what agents do, not toward hardening the UI itself as a data exposure surface.

### RDL_2 - Data Persistence and Caching
- Strength score: 1

The Google framework's memory security guidance notes that memory implementations must ensure strict isolation between users and between different contexts, and that users need transparency and control over agent memory. These principles are nominally relevant to session persistence serialization (RDL_2_1) and cached response sensitivity (RDL_2_2). However, the framework does not address caching architectures, CDN edge server retention, composite cache key security across multi-agent response aggregation, or distributed undo buffer coordination across agents (RDL_2_3). The GDPR right-to-erasure conflict with distributed undo and the specific mechanics of browser cache, CDN, and application-layer sensitive data survival beyond session termination are not addressed. The memory isolation principle provides conceptual relevance but no operational control for these persistence and caching vulnerabilities.

### RDL_3 - Streaming and Token-Level Leakage
- Strength score: 1

The Google framework does not address streaming output patterns. The framework's output sanitization guidance (XSS prevention) and prompt injection structural defenses could nominally apply to streaming handoffs as one point where injection could be detected, but the framework provides no guidance on mid-generation redaction failures where early tokens are already in UI buffers before sensitivity is recognized (RDL_3_1), tokenization side channels enabling context window inference (RDL_3_3), streaming output length correlation attacks (RDL_3_4), streaming state exposure during iterative multi-agent cycles (RDL_3_5), or streaming cache covert exfiltration channels (RDL_3_7). These are inference-time and streaming-protocol-level leakage mechanisms outside the framework's scope.

### RDL_4 - Search and Recall
- Strength score: 2

The framework directly addresses memory isolation as a core control: memory "must ensure strict isolation between users and potentially between different contexts for the same user to prevent contamination." This isolation principle is directly applicable to the probabilistic recall via conversation search threat (RDL_4_1), where semantic search indexes span all agents' histories without security boundaries, causing cross-domain content surfacing. The framework also requires user transparency and control over memory. However, the framework does not provide specific controls for access-controlled search indexes, semantic similarity boundary enforcement across agent domains, or systematic adversarial query probing to surface high-sensitivity content through semantic association. The memory isolation principle provides relevant guidance but lacks the search-index-level specificity needed to fully address this category.

### RDL_5 - Attribution and Observability
- Strength score: 2

Principle 3 requires robust logging of agent inputs, tools invoked, parameters, outputs, and reasoning steps — which is precisely what creates the multi-agent attribution intelligence leakage described here. The framework explicitly requires this logging for security purposes, but the threats in RDL_5 arise from the logs themselves becoming high-value targets. The framework's AAA infrastructure requirement implies access controls on log stores, providing partial mitigation for unauthorized log access. However, the framework does not address the aggregated intelligence value of multi-agent attribution logs revealing organizational hierarchy and agent specialization patterns (RDL_5_1), framework-dependent log aggregation enabling architectural reconstruction (RDL_5_2), error classification pattern reconnaissance (RDL_5_3), or latency measurement information leakage (RDL_5_4). The framework mandates observability for security, creating an inherent tension with the confidentiality of the observability data itself that is not resolved.

### RDL_7 - Tool Invocation and Function Calling
- Strength score: 2

Principle 2's AAA infrastructure and scoped credentials are directly relevant to controlling who can access tool invocation logs and function calling JSON. The framework explicitly requires that action logs capture tools invoked and parameters — which is both the recommended control and the source of the leakage risk in RDL_7_1 and RDL_7_8 (parameter values containing API keys and PII leaking through centralized logs). The framework's tool authentication and authorization guidance provides partial mitigation for unauthorized log access. Structural prompt conventions and input stream separation are relevant to function calling JSON persisting in shared context windows (RDL_7_2). However, the framework does not address tool error message detail reduction (RDL_7_3), tool selection reasoning as cognitive state leakage (RDL_7_4), function description enumeration revealing organizational capabilities (RDL_7_5), audit trail poisoning by compromised agents (RDL_7_6), or tool success rate differential as targeting intelligence (RDL_7_7). The framework's logging mandate creates the very log aggregation risk that RDL_7_10 describes.

### RDL_9 - Multi-Agent Memory and State
- Strength score: 2

The framework directly addresses shared memory as an attack vector: memory isolation between users and contexts is required, and persistent injection via memory is specifically called out as a concern. These controls are relevant to cross-agent reflection memory leakage (RDL_9_1), ReAct trace persistence enabling forensic reconstruction (RDL_9_2), and shared working-memory blackboard access pattern analysis (RDL_9_4). The framework's user transparency and control over memory also addresses the monitoring dashboard correlation leakage (RDL_9_5) at a high level. However, distributed tracing cross-tenant correlation via timing metadata (RDL_9_3) and false sharing timing attacks on blackboard memory locking (RDL_9_4) are lower-level infrastructure security concerns not addressed by the application-layer framework. The memory isolation guidance is the most directly applicable control in this section.

### RDL_10 - Reasoning and CoT Traces
- Strength score: 2

The Google framework explicitly requires logging of reasoning steps as part of Principle 3, making CoT trace storage a recommended practice — which is the origin of most leakage risks in this subsubsection. The framework's memory isolation requirement is directly relevant to cross-agent reasoning context leakage (RDL_10_4) and memory consolidation leakage (RDL_10_7), where one agent's reasoning trace flows to other agents through shared memory. Prompt injection defenses using structural separation are relevant to CoT traces that embed sensitive context being used as input to downstream agents (RDL_10_4). However, the framework does not address CoT trace memory query leakage (RDL_10_1), intermediate step verbosity enabling data reconstruction (RDL_10_2), reasoning-embedded secrets in shared traces (RDL_10_3), lookahead computation information leakage via tree state observation (RDL_10_5), preserved reasoning path long-lived leakage (RDL_10_6), or probabilistic recall of intermediate self-consistency chains (RDL_10_8). The framework creates the logging requirement without providing controls for the sensitivity of the logged content.

### RDL_11 - Multimodal and Input Processing
- Strength score: 1

The Google framework does not address multimodal content processing. The framework's general output sanitization guidance and memory isolation principles provide no specific controls for vision model intermediate representation leakage (RDL_11_2), audio embedding privacy through speaker characteristic encoding (RDL_11_3), chart linearization producing machine-readable sensitive numerical data (RDL_11_4), or multimodal embedding inversion attacks enabling image reconstruction from vector stores (RDL_11_5). The large context window expansion that multimodal RAG creates (RDL_11_1) is noted as increasing leakage surfaces, but the framework's guidance was written for text-based agent interactions. No applicable controls exist for these multimodal-specific leakage channels.

### RDL_12 - Error Handling and Graceful Degradation
- Strength score: 1

The framework's Principle 3 observability requirement implies error logging as part of audit trails, but provides no guidance on controlling the sensitivity of error message content. The framework does not address error message logging as a data repository (RDL_12_1), retry attempt logging exposing intermediate states (RDL_12_2), fallback provider log inconsistency creating weaker-security logging streams (RDL_12_3), graceful degradation state logs disclosing architecture (RDL_12_4), or circuit breaker state change logging enabling infrastructure reconnaissance (RDL_12_5). These are operational logging hygiene and error-handling design concerns that fall outside the framework's application-level security guidance. The framework's AAA infrastructure concept implies access controls on log stores but does not address content minimization or redaction in error conditions.

### RDL_13 - Evaluation and Testing Leakage
- Strength score: 1

The Google framework's assurance activities (regression testing, red teams, user feedback, external researchers) address evaluation as a security improvement mechanism, not as a leakage surface. The framework provides no guidance on protecting evaluation metric logs from becoming system fingerprints (RDL_13_1), baseline comparison context windows exposing historical user queries (RDL_13_2), evaluation artifact storage access controls (RDL_13_4), or benchmark result leakage through approval logs (RDL_13_9). Evaluation pipeline security — protecting the data that flows through testing and benchmarking infrastructure — is not within the framework's scope. The assurance activities are described as security inputs, not as potential attack surfaces requiring protection.

### RDL_14 - Feedback and Testing
- Strength score: 1

The Google framework explicitly mentions user feedback as a continuous assurance mechanism for improving agent security, but treats feedback as an input to defense rather than as a potential leakage vector. The specific threats — user feedback logs revealing systematic failure modes enabling targeted compromise (RDL_14_1), and A/B test results enabling attacker identification of more-vulnerable agent variants (RDL_14_2) — are not addressed. The framework's AAA infrastructure implies access controls on feedback storage, but no content protection or aggregation risk guidance is provided for feedback and testing data stores.

### RDL_15 - Evaluation Metric Exploitation
- Strength score: 1

The Google framework does not address evaluation metric design or benchmark gaming. The threats in this subsubsection — exact match metric exploitation through semantic paraphrasing (RDL_15_1), joint metric evasion via fabricated supporting facts (RDL_15_2), probabilistic non-deterministic attack evasion of Pass@K metrics (RDL_15_3), and milestone scoring threshold manipulation for partial credit exploitation (RDL_15_4) — concern adversarial manipulation of evaluation methodology rather than runtime agent behavior. The framework's assurance activities note regression testing and red teams, which could conceptually surface some of these evasion patterns, but no specific guidance on metric robustness or anti-gaming evaluation design is provided.

### RDL_16 - Parameter and Configuration Leakage
- Strength score: 1

The Google framework does not address inference parameter configuration or the side channels created by performance tuning. Context window size differentials creating differentiable log volumes (RDL_16_1), cost-optimized model routing creating observable token consumption patterns that reveal query sensitivity (RDL_16_2), and latency profile fingerprinting from parameter optimization (RDL_16_3) are operational tuning side channels outside the framework's scope. Principle 3's logging requirement could surface these patterns but provides no mitigation. No applicable framework controls exist for configuration leakage through performance-tuning artifacts.

### RDL_17 - Prompt Injection and Few-Shot Attacks
- Strength score: 2

The Google framework directly addresses prompt injection as a primary threat and proposes structural defenses (delimiters, role tagging to distinguish trusted instructions from untrusted inputs) and Layer 2 guard model classifiers to detect malicious instruction patterns. These defenses are relevant to demonstration output format injection (RDL_17_1) where instructions are embedded in few-shot examples, and to few-shot demonstration contamination with leaked sensitive data (RDL_17_3) where adversarial prompts trigger probabilistic recall. Input stream separation is specifically designed to prevent untrusted data (including demonstration examples) from being treated as trusted instructions. However, generation pattern injection through biased few-shot examples causing learned confidence bias (RDL_17_2) is a model-behavior learning concern rather than a structural injection attack, and the framework's structural defenses do not address it. The prompt injection guidance provides the most direct and applicable coverage in this subsubsection.

### RDL_18 - Distributed Tracing
- Strength score: 1

The Google framework does not address distributed tracing infrastructure. The framework's Principle 3 logging requirement implies that distributed tracing is a recommended practice, but provides no guidance on protecting trace correlation IDs and span data from becoming workflow leakage vectors (RDL_18_1). Response schema validation log side channels revealing agent configuration differences (RDL_18_2) are also outside scope. The framework's AAA infrastructure concept implies that access to tracing backends should be controlled, but no specific controls for trace data sensitivity, retention policies, or cross-tenant trace isolation are provided.

### RDL_19 - Execution and Orchestration
- Strength score: 1

The Google framework's Principle 3 requires logging of agent actions and reasoning, which directly creates the execution path disclosure risks described here. Expanded error details from one agent disclosing another agent's execution paths through shared error handling (RDL_19_1) is a multi-agent error propagation concern not addressed by the framework. Parameter provenance tracking loss creating attribution gaps (RDL_19_2) that enable downstream poisoning without visibility into origin is relevant to the framework's trusted vs. untrusted input distinction (Principle 1), but the framework does not operationalize cross-agent parameter provenance tracking or serialization integrity for handoff data. No specific controls exist for these orchestration-level leakage patterns.

### RDL_20 - Multi-Agent Reasoning
- Strength score: 2

The Google framework's Principle 1 (well-defined human controllers with trusted vs. untrusted input distinction) and Principle 3 (observable planning and reasoning) are partially relevant here. Orchestrator reasoning quality cascading effects (RDL_20_1) are addressed by the framework's plan analysis models that evaluate proposed plans for undesirable outcomes before execution, and Layer 1 policy engines that operate outside AI reasoning to intercept cascading malicious decisions. The transparency paradox in supervisor delegation reasoning (RDL_20_2) reflects a genuine tension the framework does not resolve — it mandates transparency while acknowledging that transparency can be exploited. Cross-boundary reasoning coherence validation (RDL_20_3) and consensus reasoning exploitation (RDL_20_4) are systemic multi-agent coordination problems for which the framework provides no specific mechanism.

### RDL_21 - Efficiency and Cost Attribution
- Strength score: 1

The Google framework does not address efficiency monitoring, cost attribution systems, or the side channels they create. Efficiency logs leaking sensitive operational patterns through token consumption spikes (RDL_21_1), cost attribution metadata revealing data sensitivity from operational cost differentials (RDL_21_2), cache performance metrics indicating high-value target frequency (RDL_21_3), latency anomaly correlation with payload size (RDL_21_4), and efficiency dashboard exposure of system internals (RDL_21_5) are monitoring-plane side channels entirely outside the framework's scope. Principle 3's logging requirement creates some of these channels without providing controls for their confidentiality.

### RDL_22 - Infrastructure and Deployment
- Strength score: 1

The Google framework operates at the application and architecture layer and provides no guidance for the infrastructure-level leakage channels in this subsubsection. Message queue header correlation enabling workflow reconstruction (RDL_22_1), vector database embedding exposure enabling approximate text recovery (RDL_22_2), Prometheus metric label cardinality exploitation (RDL_22_3), API gateway log correlation for user activity reconstruction (RDL_22_4), MLflow versioning history revealing security issues (RDL_22_5), dead-letter queue content forensics (RDL_22_6), and centralized logging pipeline compromise enabling cross-agent exfiltration (RDL_22_7) are all infrastructure deployment security concerns. The framework's AAA infrastructure concept nominally implies access controls on these systems, but no specific guidance is provided for any of them.

### RDL_23 - Containerization Security
- Strength score: 1

The Google framework explicitly mentions sandboxing as a control under Principle 2, asking "What safeguards (such as sandboxing policies) prevent agent actions from improperly exposing high-privilege information or capabilities?" This provides nominal relevance to containerization security. ### RND_1 - UI/Streaming and Real-Time Generation
