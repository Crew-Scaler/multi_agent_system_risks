# Risk Management Analysis: NSA AI Data Security Framework

**Framework:** AI Data Security: Best Practices for Securing Data Used to Train & Operate AI Systems
**Authors:** NSA Artificial Intelligence Security Center (AISC), CISA, FBI, ASD's ACSC, NCSC-NZ, NCSC-UK
**Document:** Cybersecurity Information Sheet (CSI), May 2025, Ver. 1.0

**Framework Summary:** This joint cybersecurity information sheet provides guidance on securing data used in AI and machine learning systems across all six NIST AI RMF lifecycle stages. Its ten best practices focus on: (1) sourcing reliable data and tracking provenance via cryptographically signed append-only ledgers, (2) verifying data integrity with checksums and cryptographic hashes, (3) employing quantum-resistant digital signatures for training data, RLHF, and fine-tuning datasets, (4) leveraging trusted infrastructure with Zero Trust architecture and secure enclaves, (5) classifying data and enforcing access controls, (6) encrypting data at rest, in transit, and during processing (AES-256/TLS/post-quantum), (7) storing data in FIPS 140-3 certified devices, (8) applying privacy-preserving techniques (data masking, differential privacy, federated learning, secure multi-party computation), (9) deleting data securely per NIST SP 800-88, and (10) conducting ongoing risk assessments using NIST RMF and AI RMF. The framework addresses three specific risk areas in depth: data supply chain security (split-view poisoning, frontrunning poisoning, web-crawled datasets), maliciously modified data (adversarial ML attacks, statistical bias, data poisoning, deduplication failures), and data drift (distribution shift detection and mitigation). The framework does not address UI/UX design, multi-agent trust exploitation, prompt injection, tool invocation security, agentic workflow attacks, hardware-level attacks, or distributed systems concurrency concerns.

## RATC - Agent–tool coupling as "policy‑level remote code execution"

### RATC_1 - UI/UX Abstraction and Visibility Gaps - Strength score: 1

The NSA AI Data Security framework is scoped exclusively to data integrity, supply chain security, and data drift, and provides no controls addressing UI/UX design, approval workflow visualization, or how tool invocations are surfaced to human reviewers. The risks described in this subsubsection—approval workflow risk calibration failure through tool abstraction (RATC_1_1), progressive disclosure concealment of tool chain complexity (RATC_1_2), tool visibility gaps across UI paradigms (RATC_1_3), insufficient tool risk differentiation in multi-agent dashboards (RATC_1_4), context-driven tool selection without user awareness (RATC_1_5), and trace visualization abstraction concealing tool chain complexity (RATC_1_6)—all involve how humans perceive and interact with multi-agent systems through visual interfaces, not how training data is secured or provenance tracked. The framework's access control guidance (BP5) is tangentially relevant in that classifying data by sensitivity informs what operations should require approval, but this does not translate into controls over how approval workflows visually represent or abstract the risks of those operations to human reviewers. These risks require dedicated UI/UX security engineering and human-computer interaction controls beyond this framework's scope.

### RATC_2 - Approval Workflow and User Attention Vulnerabilities - Strength score: 1

The NSA AI Data Security framework provides no applicable controls for approval fatigue exploitation (RATC_2_1), keyboard shortcut hijacking in approval workflows (RATC_2_2), or time-based default exploits for critical decisions (RATC_2_3). These risks involve human factors engineering, browser security, and workflow timing attacks that are unrelated to training data integrity, data supply chain security, or data drift detection. The framework's guidance on ongoing risk assessments (BP10, using NIST RMF) would identify human-in-the-loop approval workflow vulnerabilities as a risk category requiring management, but the framework provides no specific controls for the attack vectors described. The core vulnerability—that adversaries can manipulate human attention and timing to obtain approvals for dangerous tool invocations—requires UI security controls, rate limiting on approval workflows, and mandatory review periods that are entirely outside this data-focused framework.

### RATC_3 - Tool Overload, Parameter Handling, and Selection Errors - Strength score: 1

The NSA AI Data Security framework does not address tool overload privilege escalation (RATC_3_1), tool argument injection and parameter extraction errors across agent trust boundaries (RATC_3_2), tool visibility gaps in inline suggestion patterns (RATC_3_3), or tool provenance obscuration in chat interface outputs (RATC_3_4). These risks involve agent operational logic, multi-agent parameter passing, and UI transparency rather than data security. The framework's data integrity controls (BP2 checksums, BP3 digital signatures) are conceptually relevant to ensuring parameter values have not been tampered with at rest, but the framework provides no controls applicable to real-time agentic tool invocation parameter validation during agent operation, nor to the probabilistic LLM generation process that enables argument injection when one agent constructs parameters from untrusted data for another agent to execute. The tool chain complexity and provenance visibility risks in chat interfaces require dedicated agentic observability and audit controls.

### RATC_4 - Confidence Manipulation and Tool Authorization - Strength score: 1

The NSA AI Data Security framework does not address confidence threshold manipulation through adversarial prompt injection (RATC_4_1) or tool authorization scope confusion across agent specializations (RATC_4_2). Confidence score manipulation exploits how the model evaluates evidence and synthesizes reasoning paths—a cognitive vulnerability in the deployed model that cannot be addressed by securing training data supply chains or detecting data drift. The framework's access control guidance (BP5, data classification and access controls) conceptually aligns with least-privilege authorization principles, but is framed around protecting data assets rather than scoping agent tool invocation permissions. The precondition validation bypass risk, where one agent verifies prerequisites but downstream agents skip rechecking under transitive trust assumptions, requires agent-level authorization architecture controls rather than data security measures.

### RATC_6 - Tool Metadata Poisoning Across Registries and Discovery - Strength score: 2

The NSA AI Data Security framework provides moderate indirect mitigation for tool registry metadata poisoning because tool descriptions, parameter schemas, and few-shot examples stored in centralized registries and vector databases are fundamentally data assets subject to the framework's integrity and provenance controls. BP2 (checksums and cryptographic hashes to detect unauthorized changes), BP3 (digital signatures authenticating trusted data revisions), and BP1 (provenance tracking with cryptographically signed append-only ledger) could be applied to secure tool registry contents against the metadata poisoning attacks described in RATC_6_1 and RATC_6_2. The specific risk of vector database metadata poisoning causing semantic search to retrieve wrong tools corresponds directly to the data integrity problem the framework addresses. Similarly, few-shot parameter examples in tool descriptions embedding malicious instructions represent a data supply chain threat analogous to the training dataset poisoning scenarios the framework targets. However, the framework does not specifically address tool registry security scenarios—its guidance is framed around training datasets and operational data—and provides no guidance for framework abstraction leakage (RATC_6_3) or the multi-framework tool selection poisoning attack surface scaling with framework interaction pairs.

### RATC_8 - Advanced Tool Invocation Patterns and Inference - Strength score: 1

The NSA AI Data Security framework provides no applicable controls for tool chain exploitation through multi-agent replanning loops (RATC_8_1), tool result injection and observation manipulation in ReAct-style architectures (RATC_8_2), or tool schema variation across cross-framework boundaries (RATC_8_3). These are operational agentic attack vectors involving real-time manipulation of agent reasoning streams, observation pipelines, and framework interoperability layers that operate entirely outside the data security domain. The framework's data integrity controls (BP2, BP3) are not applicable to securing real-time agent observation streams, and the framework provides no guidance for the replay, injection, and schema translation attack vectors that exploit gaps in multi-agent tool invocation pipelines. Streaming attacks enabling progressive escalation injection compounding across orchestration layers represent a distinctly operational threat category beyond this framework's training and supply chain focus.

### RATC_9 - Web and Multimodal Tool Exploitation - Strength score: 2

The NSA AI Data Security framework provides relevant indirect coverage for the data-facing aspects of web and multimodal tool exploitation. The framework directly addresses risks arising from web-crawled and web-scale datasets, including split-view poisoning (expired domain attacks enabling malicious content substitution) and frontrunning poisoning (malicious edits ahead of dataset snapshots), with mitigations including cryptographic hash verification of raw data, content credentials, dataset certification requirements, and periodic curator verification. These controls reduce the risk of malicious web content being incorporated into RAG knowledge bases that multimodal retrieval agents access (RATC_9_2). The framework's maliciously modified data guidance—covering anomaly detection, data sanitization, and secure training pipelines—is applicable to the multimodal content poisoning scenarios where retrieval agents select malicious documents. However, the framework does not address real-time web scraping injection attacks (RATC_9_1) where adversaries inject instructions into live web content during agent operation, nor cross-modal tool parameter injection where multiple modalities contribute parameter segments without sanitization boundaries, as these operational attack vectors occur independently of the static training data supply chain.

### RATC_10 - Efficiency Optimization and Resource Constraints - Strength score: 1

The NSA AI Data Security framework does not address token allocation vulnerabilities forcing truncation of safety-critical tool descriptions (RATC_10_1), iteration budget exploitation where crafted tool sequences succeed only against high-budget agents, temperature tuning attacks separating actual accuracy from confidence visibility, or model selection diversity creating policy inconsistency (RATC_10_2). These risks involve model configuration parameters, agent operational resource budgets, and inference-time behavior that are entirely outside the data security domain. The framework's guidance covers securing data used to build and operate AI systems but provides no controls applicable to how adversaries exploit computational resource constraints to cause safety-critical context truncation or exploit confidence-accuracy decoupling. These attack vectors require model evaluation, inference infrastructure, and agent configuration controls.

### RATC_11 - Tool Execution Infrastructure and Orchestration - Strength score: 1

The NSA AI Data Security framework's trusted infrastructure guidance (BP4, Zero Trust architecture and secure enclaves) and access controls (BP5) provide tangential relevance to infrastructure orchestration security. Zero Trust principles applied to data workflows imply consistent verification that would, if extended across the full infrastructure stack, reduce the blast radius of API gateway routing compromises (RATC_11_1) and Kubernetes container security context exploitation (RATC_11_2). However, the framework explicitly scopes its guidance to data workflows and does not address service mesh attack surfaces, API gateway routing policy manipulation enabling tool call redirection to attacker-controlled services, Kubernetes pod security context privilege escalation, persistent volume claim vulnerabilities, sidecar proxy interception, or init container configuration injection in multi-agent microservice deployments. These infrastructure orchestration attack vectors require dedicated infrastructure security controls and Kubernetes security hardening beyond this data-focused framework.

### RATC_12 - Distributed and Hardware-Level Attacks - Strength score: 1

The NSA AI Data Security framework does not address tensor parallelism GPU communication interception (RATC_12_1), KV cache poisoning through multi-user inference on shared GPUs, or quantization calibration dataset poisoning causing systematic tool selection bias (RATC_12_2). These attacks target the physical computation infrastructure beneath the data layer—inter-GPU communication channels, shared inference memory, and hardware quantization pipelines. The framework focuses exclusively on protecting data integrity and supply chains at the logical level and provides no controls applicable to GPU memory isolation failures, unencrypted inter-GPU communication attack surfaces, or quantization pipeline bias injection. The framework's secure storage guidance (BP7, FIPS 140-3 certified devices) addresses data storage security but does not extend to the inference-time hardware attack vectors described here.

### RATC_14 - Reasoning and Planning Vulnerabilities - Strength score: 1

The NSA AI Data Security framework does not address chain-of-thought reasoning quality failures where agents make unsupported leaps (RATC_14_1), poisoned reasoning trace injection propagating malicious tool preferences through shared memory, self-consistency and majority voting manipulation through manipulated sampling paths (RATC_14_2), or hierarchical task network planning vulnerabilities causing tool visibility fragmentation (RATC_14_3). While the framework's BP3 (digital signatures for RLHF data) is conceptually relevant since poisoned RLHF reward signals could embed malicious reasoning trace patterns that later manifest as biased chain-of-thought outputs, the specific attack vectors described involve real-time manipulation of deployed agent reasoning rather than training data corruption. The framework provides no controls for securing the internal reasoning and planning processes of deployed agents or for protecting shared reasoning memory from cross-agent trace poisoning during operation.

### RATC_15 - Episodic Memory and Learning - Strength score: 1

The NSA AI Data Security framework does not address episodic memory poisoning where attackers record malicious tool sequences as successful resolutions (RATC_15_1), trajectory abstraction converting poisoned episodes into dangerous procedural workflows, or cross-agent tool coordination compromise through shared episodic memory enabling one agent's compromise to propagate as learned policy across the agent fleet. Episodic memory is an operational runtime system where agents store and retrieve experience records during deployment, not a training dataset subject to the framework's supply chain and integrity controls. The framework's guidance on securing training data (including RLHF datasets) does not extend to the dynamic operational memory systems agents use during inference. Protecting episodic memory requires runtime memory access controls, integrity verification for stored episodes, and anomaly detection in retrieved action sequences.

### RATC_16 - Semantic Memory and RAG - Strength score: 2

The NSA AI Data Security framework provides moderate relevant coverage for RAG pipeline and knowledge base poisoning because the knowledge base content that retrieval agents access is fundamentally data subject to the framework's integrity, provenance, and supply chain controls. BP2 (checksums and cryptographic hashes verifying that data has not been altered) and BP3 (digital signatures authenticating trusted data revisions) are directly applicable to verifying that knowledge base documents have not been tampered with to embed malicious tool-invocation instructions (RATC_16_1). BP1 (cryptographically signed append-only provenance ledger) enables tracing the origin and modification history of documents in knowledge graphs, helping detect relationship poisoning causing unsafe tool combinations to appear recommended. The framework's maliciously modified data guidance—covering anomaly detection, data sanitization, and secure data pipelines—addresses the statistical bias and deduplication failure risks relevant to knowledge base content quality. However, the framework does not specifically address query rewriting instruction injection, knowledge graph relationship-level poisoning as a distinct threat, or knowledge base staleness management requiring operational data lifecycle controls beyond supply chain verification.

### RATC_17 - Utility Functions and Decision Logic - Strength score: 1

The NSA AI Data Security framework does not address utility-weighted tool selection manipulation through outcome assumption injection (RATC_17_1), sequential expected utility miscalculation through misrepresentation of available future options, or multi-objective utility function weight poisoning balancing competing objectives. These vulnerabilities target the internal decision logic architecture of agent systems—how agents mathematically evaluate and rank potential actions—which operates at the model and agent framework level rather than the data pipeline level. The framework's data provenance and integrity controls are not applicable to securing utility calculation inputs during real-time agent decision-making. These decision logic integrity risks require agent-level controls such as utility function auditing, sandboxed evaluation environments, and tamper-resistant weight stores.

### RATC_18 - Rule-Based and Knowledge-Engineered Systems - Strength score: 1

The NSA AI Data Security framework does not address rule-based authorization bypass through specificity hierarchy injection (RATC_18_1), forward chaining rule exploitation enabling unauthorized tool authorization through injected facts, certainty factor manipulation causing risk assessment bypass, or lexicographic heuristic objective reordering causing wrong tool selections. These are integrity attacks targeting rule system logic and fact bases rather than training data supply chains. While the framework's data integrity controls could theoretically be applied to secure rule database files if treated as data assets requiring hash verification and digital signatures, the framework provides no specific guidance for these attack vectors, and the dynamic nature of forward chaining inference during operation makes static data integrity controls insufficient to prevent real-time rule injection attacks.

### RATC_19 - Learning and Reinforcement Learning - Strength score: 2

The NSA AI Data Security framework provides moderate relevant coverage for RL tool selection policy hijacking through its explicit guidance on securing RLHF training data. BP3 specifically calls out adopting quantum-resistant digital signatures to authenticate datasets used during AI model training, fine-tuning, alignment, and reinforcement learning from human feedback, directly addressing the risk that poisoned reward signals or maliciously crafted expert demonstrations corrupt RL-trained agent policies (RATC_19_1). The framework's data integrity and provenance controls (BP1, BP2) applied to RL training datasets, demonstration datasets, and replay buffer snapshots would reduce the attack surface for curriculum learning poisoning and imitation learning trajectory poisoning during the offline training phase. However, the framework does not address runtime RL attacks such as multi-agent RL shared replay buffer poisoning during live online training, actor-critic cross-agent critic value estimation manipulation in deployed systems, or proximal policy optimization trust region manipulation iteratively converging deployed agent policies toward dangerous attractors. These operational RL attack vectors require runtime monitoring and policy integrity controls beyond training data security.

### RATC_21 - Parallel Retrieval Race Conditions in Multi-Agent Query Decomposition Systems - Strength score: 1

The NSA AI Data Security framework does not address parallel retrieval race conditions in shared indexes, deduplication races where concurrent agents mark documents as seen producing inconsistent results, timeout-triggered retrieval cascade failures overwhelming connection pools, or multi-stage caching temporal inconsistency from mismatched TTL expiration (RATC_21). The framework's supply chain guidance (dataset verification, digital signatures, registry credential security) provides marginal coverage of the container supply chain attack vector mentioned—registry credential compromise enabling fleet-wide poisoning during automated deployment—but this represents only a small component of the broader race condition and distributed consistency problem. The core parallel retrieval consistency risks are distributed systems engineering concerns requiring optimistic concurrency control, cache coherence protocols, and circuit breaker patterns that are entirely outside this data-focused framework's scope.

### RATC_22 - Multi-Stage Pipeline Result Caching Race Enabling Multi-Agent Consistency Failures - Strength score: 1

The NSA AI Data Security framework does not address multi-stage pipeline caching race conditions from independent TTL expiration, event-based cache invalidation propagation delays causing knowledge state inconsistency, write races and hash collision attacks in shared caching infrastructure, or cache warming inconsistency (RATC_22). The framework's supply chain guidance provides tangential coverage for the registry credential compromise and typosquatting attack vectors enabling fleet-wide backdoor deployment through GitOps automation, as its dependency verification and supply chain security principles are applicable. However, this covers only a narrow slice of the described threat, and the core pipeline caching consistency and race condition attack surfaces require distributed systems reliability engineering well beyond the data security framework's training and supply chain focus. The tensor parallelism and profiling telemetry exposure vectors are hardware and observability concerns entirely outside the framework's scope.

### RATC_27 - MIG Instance Co-Location Creating Shared Physical GPU Failure Propagation - Strength score: 1

The NSA AI Data Security framework does not address Multi-Instance GPU (MIG) partition isolation failures, shared physical hardware dependency attacks through power regulator or thermal management exploitation, PCIe interface vulnerabilities, GPU firmware exploits enabling cross-partition memory access, or GPU driver stack single-point-of-failure risks from CUDA operation triggered driver bugs (RATC_27). These are hardware-level infrastructure concerns targeting the physical computation substrate beneath the data and model layers. The framework's trusted infrastructure guidance (BP4, Zero Trust architecture and secure enclaves) advocates for isolation of sensitive operations but does not extend to GPU virtualization isolation security or hardware partition integrity verification. This represents an honest gap: the framework was not designed to address physical hardware attack surfaces in shared GPU infrastructure.

### RATC_29 - Shared Workflow State Version Conflict Amplification Through Concurrent Write Flooding - Strength score: 1

The NSA AI Data Security framework does not address shared workflow state version conflicts arising from optimistic locking in concurrent multi-agent coordination systems (RATC_29), retry storm cascades that intensify contention through positive feedback loops, delayed writes extending version epochs increasing stale reads, or targeted collision bursts exhausting retry budgets causing workflow failures without completion. These are distributed systems concurrency problems in operational workflow coordination infrastructure, entirely outside the data security framework's scope of training data integrity and supply chain security. The framework's data integrity guidance (checksums, hashes, append-only ledgers) addresses tamper detection for stored data, not concurrent write coordination semantics or optimistic locking race condition management in live multi-agent deployments.

### RATC_30 - Guardrail Bypass Through Infrastructure Failure Injection - Strength score: 1

The NSA AI Data Security framework does not address guardrail infrastructure timeout exploitation where adversary-induced processing delays cause safety checks to be skipped (RATC_30), GPU inference delay attacks causing validation timeouts, network flooding against centralized PII detection or policy rule evaluation endpoints, or resource exhaustion attacks causing fleet-wide guardrail bypass when agents fall back to delivering unvalidated responses. The framework's ongoing risk assessment guidance (BP10, NIST RMF) would identify guardrail infrastructure availability as a risk category requiring management, but the framework provides no specific controls for guardrail infrastructure resilience, fault tolerance design, or defense against targeted denial-of-service attacks against validation endpoints. These availability and reliability concerns require infrastructure engineering controls—redundant validation pipelines, circuit breakers with fail-safe defaults, and rate limiting on validation endpoints—that are beyond this data security framework's guidance.

## RDL - New data‑leakage channels via large contexts, logs, and probabilistic recall

### RDL_1 - UI/UX Patterns - Strength score: 1

The NSA AI Data Security framework does not address the UI/UX data leakage channels described in this subsubsection, including chat interface conversation history accumulating sensitive data as persistent repositories (RDL_1_1), progressive disclosure debug views exposing credentials and system internals (RDL_1_2), session persistence enabling cross-security-boundary data retention (RDL_1_3), command palette logs recording sensitive operation sequences (RDL_1_4), multi-agent dashboards exposing compartmentalized data correlations (RDL_1_5), reference links as indirect information disclosure (RDL_1_6), approval workflow audit trails as organizational intelligence (RDL_1_8), approval evidence links bypassing role-based permissions (RDL_1_9), refinement iteration histories exposing decision logic (RDL_1_10), and accessibility feature (alt text, ARIA labels) leakage (RDL_1_11). The framework's data classification guidance (BP5) provides a conceptual foundation by requiring sensitivity-based categorization that, if extended to UI data handling policies, could inform how dashboard elements and session data are protected. Privacy-preserving data masking (BP8) could prevent PII exposure in UI components if applied to displayed data. However, the framework focuses on data pipeline and supply chain security rather than UI component design, session storage architecture, or browser-side data exposure controls, leaving the broad range of UI/UX-driven leakage vectors without specific applicable guidance.

### RDL_2 - Data Persistence and Caching - Strength score: 2

The NSA AI Data Security framework provides moderate applicable coverage for this subsubsection's persistence and caching data leakage risks. BP6 (encrypt data at rest using AES-256 and in transit using TLS) directly applies to securing session storage, browser caches, and application-layer caching that persist multi-agent conversation data beyond intended lifetimes (RDL_2_2). BP7 (store data in FIPS 140-3 certified devices) applies to persistent session state stores containing aggregated sensitive data from multiple agents (RDL_2_1). Most directly applicable is BP9 (secure deletion using cryptographic erase or data overwrite per NIST SP 800-88), which addresses the undo buffer retention problem (RDL_2_3) where multi-agent distributed undo creates coordination complexity for synchronized buffer clearing—the framework explicitly recommends secure deletion methods to ensure data users believe has been removed is truly irrecoverable. BP5 (data classification with sensitivity-based access controls) provides guidance for enforcing appropriate access controls on cached and persisted multi-agent session data. However, the framework does not specifically address the multi-agent cache key isolation problem that causes cross-user data exposure, or the distributed undo coordination complexity across heterogeneous agents with inconsistent retention policies.

### RDL_3 - Streaming and Token-Level Leakage - Strength score: 1

The NSA AI Data Security framework provides minimal applicable coverage for streaming and token-level leakage. BP6 (TLS with AES-256 or post-quantum encryption for data in transit) applies to protecting network-layer streaming transmissions from interception, reducing the RATC_3_1 streaming token-by-token leakage risk at the transport layer. However, the framework does not address the streaming-specific vulnerabilities that operate above the encryption layer: mid-generation redaction failure where early tokens are already in UI buffers before post-processing redaction can apply (RDL_3_1), tokenization pattern inference enabling context window and system architecture mapping (RDL_3_3), output length correlation enabling session reconstruction without content access (RDL_3_4), streaming state update exposure across multi-agent cycles (RDL_3_5), intermediate state visibility through partial responses (RDL_3_6), or streaming cache side-channel exfiltration (RDL_3_7). These streaming-specific attack vectors require application-layer streaming security controls—pre-stream content screening, token-level redaction pipelines, streaming response randomization, and cache isolation—that are entirely outside this data security framework's scope.

### RDL_4 - Search and Recall - Strength score: 1

The NSA AI Data Security framework does not address probabilistic recall through conversation search (RDL_4_1), where semantic similarity matching crosses security domain boundaries in unified multi-agent search indexes. The framework's access control guidance (BP5, data classification enabling enforcement of protection measures) is conceptually applicable—if search indexes enforced access controls aligned with data classification, semantic cross-contamination between security domains would be limited. Differential privacy techniques (BP8) could theoretically be applied to search query responses to prevent targeted probing for high-sensitivity content. However, the framework does not provide specific guidance for search index security boundary enforcement, access-controlled semantic search, or defenses against systematic adversarial probing designed to surface sensitive content through semantic association. These risks require dedicated access-controlled search infrastructure with per-query authorization enforcement beyond the framework's data pipeline focus.

### RDL_5 - Attribution and Observability - Strength score: 1

The NSA AI Data Security framework provides limited applicable coverage for attribution and observability leakage risks. BP5 (access controls) and BP6 (encryption at rest) apply to protecting log storage systems that contain multi-agent attribution metadata (RDL_5_1), ensuring that compromising log access requires defeating both access control and encryption layers. BP7 (FIPS 140-3 certified storage) applies to securing the log infrastructure hosting these high-value reconnaissance targets. However, the framework does not address the specific organizational intelligence leakage through attribution logs revealing agent ecosystem-level patterns, role distributions, and strategic priorities (RDL_5_1); framework-dependent logging aggregating cross-framework telemetry into comprehensive architecture disclosures (RDL_5_2); error classification pattern leakage enabling infrastructure fragility mapping (RDL_5_3); or latency measurement timing attacks enabling attack-timing optimization against operational systems (RDL_5_4). These observability-based intelligence gathering risks require log data minimization, telemetry access controls, and operational security practices beyond the data security framework's scope.

### RDL_7 - Tool Invocation and Function Calling - Strength score: 1

The NSA AI Data Security framework provides limited applicable coverage for tool invocation data leakage. BP8's data masking guidance is directly relevant to RDL_7_8 (parameter values containing API keys, account numbers, and PII leaking through centralized action log aggregation)—applying data masking to log pipeline ingestion would prevent sensitive parameters from persisting in queryable log repositories. BP5 (access controls) and BP6 (encryption at rest) apply generally to protecting log storage systems against unauthorized access. However, the framework does not specifically address tool invocation logging as a side channel (RDL_7_1), function calling JSON persisting sensitive parameters across agent context boundaries (RDL_7_2), implementation detail disclosure through tool error messages (RDL_7_3), tool selection reasoning as cognitive state leakage (RDL_7_4), function description enumeration via plugin registry APIs (RDL_7_5), audit trail poisoning (RDL_7_6), tool success rate differential leakage enabling targeted agent compromise (RDL_7_7), action accuracy metric operational pattern exposure (RDL_7_9), or tool execution log aggregation as a unified data exfiltration channel (RDL_7_10). These tool invocation leakage vectors require dedicated observability security controls beyond the data security framework.

### RDL_9 - Multi-Agent Memory and State - Strength score: 1

The NSA AI Data Security framework does not address multi-agent memory and state leakage channels, including reflection memory becoming unauthorized conduits across trust boundaries (RDL_9_1), ReAct reasoning trace persistence creating comprehensive attack graphs of agent topology and decision logic (RDL_9_2), distributed trace cross-tenant correlation leakage through timing analysis (RDL_9_3), blackboard shared memory access pattern reconstruction and false sharing timing attacks (RDL_9_4), or multi-agent dashboard correlation leakage through simultaneous activity patterns (RDL_9_5). The framework's encryption (BP6) and access control (BP5) guidance applies generally to protecting shared memory stores, and secure storage guidance (BP7) applies to persisted trace repositories. However, the framework does not address the cross-agent memory isolation problem, timing-based covert channels in shared memory architectures, or non-deterministic probabilistic recall that makes leakage detection difficult through traditional audits. These memory-level leakage risks require runtime memory access isolation, differential privacy in memory consolidation, and timing-resistant shared state architectures.

### RDL_10 - Reasoning and CoT Traces - Strength score: 1

The NSA AI Data Security framework does not address the chain-of-thought trace leakage channels described in this subsubsection, including CoT traces containing sensitive context leaking through memory queries (RDL_10_1), intermediate step verbosity enabling input reconstruction (RDL_10_2), reasoning-embedded secrets in shared traces (RDL_10_3), cross-agent reasoning context replication (RDL_10_4), lookahead computation branch observation leaking future plans (RDL_10_5), preserved reasoning path context as long-lived leakage vectors (RDL_10_6), memory consolidation inadvertently preserving sensitive multi-source information (RDL_10_7), intermediate trace probabilistic recall vulnerabilities (RDL_10_8), quality score metadata as side-channel organizational intelligence (RDL_10_9), and failure case logging leaking data through multi-agent learning-from-failure systems (RDL_10_10). These are operational agent reasoning architecture concerns that require runtime trace security controls—selective trace retention, trace content screening, privacy-preserving memory consolidation, and access-controlled trace storage—none of which are addressed by the data supply chain and training data integrity focus of this framework.

### RDL_11 - Multimodal and Input Processing - Strength score: 1

The NSA AI Data Security framework provides minimal applicable coverage for multimodal data leakage. BP8's differential privacy guidance has indirect relevance to protecting multimodal embeddings from inversion attacks (RDL_11_5)—adding calibrated noise to embedding representations reduces the fidelity with which attackers can reconstruct original images from stored embeddings. BP5 (access controls) and BP6 (encryption) apply to securing shared vector databases storing multimodal embeddings (RDL_11_3, RDL_11_4). However, the framework does not specifically address multimodal context window expansion creating proportionally larger leakage surfaces (RDL_11_1), vision model intermediate representation (attention maps, feature maps) leakage (RDL_11_2), audio embedding speaker identity and emotional tone privacy risks (RDL_11_3), chart linearization creating persistent leakage of sensitive numerical data in machine-readable form (RDL_11_4), or the specific attack mathematics of embedding inversion requiring differential privacy parameters calibrated to reconstruction attack capabilities. These multimodal embedding security risks require specialized vector store security and privacy-preserving embedding techniques beyond the framework's training data scope.

### RDL_12 - Error Handling and Graceful Degradation - Strength score: 1

The NSA AI Data Security framework provides limited applicable coverage for error handling data leakage. BP8 (data masking) is directly applicable to preventing sensitive data—user data, credentials, API responses—from appearing in error log entries (RDL_12_1); masking sensitive fields before error logging would reduce the attack value of error log repositories. BP5 (access controls) and BP6 (encryption at rest) apply to protecting centralized error log repositories against unauthorized access. BP7 (secure storage) applies to the error log infrastructure hosting sensitive debugging records. However, the framework does not address retry attempt logging exposing intermediate sensitive states (RDL_12_2), fallback provider logging creating inconsistent access control postures (RDL_12_3), graceful degradation state logging disclosing system architecture (RDL_12_4), or circuit breaker state transition logging as infrastructure reconnaissance enabling targeted attacks against fragile components (RDL_12_5). These error-handling leakage risks require log data minimization policies, log access governance, and error message redaction pipelines beyond the framework's scope.

### RDL_13 - Evaluation and Testing Leakage - Strength score: 1

The NSA AI Data Security framework provides limited applicable coverage for evaluation and testing leakage. BP7 (secure storage) and BP5 (access controls) apply to protecting evaluation artifact storage systems that aggregate generated reports, detailed logs, and intermediate calculations (RDL_13_4). BP6 (encryption at rest) applies to protecting stored evaluation datasets, model outputs, and reasoning traces from unauthorized access (RDL_13_5). BP8 (data masking) could prevent PII in test datasets from being exposed through evaluation dashboards (RDL_13_3). However, the framework does not address the evaluation-specific leakage vectors: metric computation logging revealing proprietary algorithms and cost structures (RDL_13_1), baseline comparison agent context window exposure of historical sensitive data (RDL_13_2), metric fingerprinting enabling evasion attack crafting (RDL_13_6), temporal evaluation result analysis inferring model update schedules (RDL_13_7), cross-validation dataset structure exposure (RDL_13_8), or benchmark result approval log leakage revealing agent failure modes (RDL_13_9). These evaluation pipeline security risks require evaluation infrastructure isolation and access controls beyond this framework's training data focus.

### RDL_14 - Feedback and Testing - Strength score: 1

The NSA AI Data Security framework does not address user feedback extraction as agent vulnerability profiling (RDL_14_1) or A/B test result leakage enabling targeted attack prediction against more vulnerable agent variants (RDL_14_2). These risks involve operational feedback and experimentation systems that collect and store agent behavioral data during production deployment.
