# Category 5.4 [NEW]: MAS Infrastructure and Orchestration Supply Chain Risks [RATC_11, RTC, RATC_23–28]

*Alignment: Expands GenAI Domain 5 with container/GPU infrastructure attack vectors from RATC_11 and fleet-level training compromise from RTC.*

**Description:** Production MAS deployments introduce supply chain risks through centralized shared infrastructure — NIM microservices, Triton inference servers, Riva ASR/TTS, shared GPU clusters, and NeMo Guardrails — that, when compromised, affect entire agent fleets simultaneously rather than individual agents.

## Frameworks

None (all risks carry [NoFW] designation)

## Key Risks

- 5.4.1 NIM Microservice Dependency Creating Fleet-Wide Single Points of Failure: Centralized NIM containers serving model inference for multiple agents create single points of failure where a compromised or degraded NIM affects all dependent agents simultaneously. [NoFW]
- 5.4.2 Triton Inference Server Request Queue Saturation for Correlated Agent Degradation: Saturating the shared Triton inference queue causes correlated latency degradation across all agents sharing the inference infrastructure. [NoFW]
- 5.4.3 Centralized NeMo Guardrails Service Saturation Enabling Fleet-Wide Output Filter Bypass: Overwhelming a centralized guardrails service forces agents to fall back to unfiltered output delivery when the guardrails service is unavailable. [NoFW]
- 5.4.4 Shared Container Host Kernel Vulnerability Enabling Fleet Compromise: A kernel vulnerability in the shared host running containerized agents can be exploited to escape container isolation and compromise all co-located agent containers. [NoFW]
- 5.4.5 MIG Instance Co-Location Creating Shared Physical GPU Failure Propagation: Although Multi-Instance GPU partitions provide hardware isolation, physical GPU failures affect all MIG instances simultaneously, creating correlated failure risk across agents sharing the same GPU. [NoFW]
- 5.4.6 Centralized ABAC Policy Engine Saturation Creating Fleet-Wide Authorization Bypass: Saturating the centralized attribute-based access control policy engine prevents authorization evaluations, causing agents to default to open or closed states depending on fail configuration. [NoFW]
- 5.4.7 RAG Pipeline and Knowledge Base Poisoning as Fleet-Wide Tool Selection Corruption: Centralized RAG knowledge bases shared across multi-agent fleets become single points of semantic control where poisoning affects all downstream agents' tool selection simultaneously. Attackers poison knowledge bases by embedding tool-invocation instructions in retrieved documents that agents treat as factual context. Semantic similarity false positive injection exploits vector similarity matching: adversarially crafted documents with high semantic similarity to operational queries are retrieved alongside legitimate content and interpreted as authoritative guidance for tool selection. Knowledge graph traversal path poisoning embeds false tool-combination recommendations in relationship edges, causing agents traversing those paths to invoke unsafe tool sequences appearing as recommended by the knowledge graph. Query rewriting instruction injection modifies optimization examples so that agents' query reformulation processes inject instructions affecting all agents sharing the query rewriting pipeline. Shared centralized knowledge bases transform per-document poisoning into fleet-wide tool selection corruption; isolated per-agent knowledge bases limit each poisoning to one agent's retrieval scope. [NoFW]
- 5.4.8 Parallel Retrieval Race Conditions in Multi-Agent Query Decomposition Systems: When a multi-agent system decomposes queries into parallel sub-queries, concurrent retrieval threads read from and write to shared indexes in overlapping windows, producing inconsistent result sets that downstream agents treat as authoritative. Deduplication race conditions cause the same malicious document to appear multiple times in merged results when Agent A's "seen" marking races with Agent B's deduplication pass, amplifying the poisoned document's influence on downstream tool selection. Async cancellation cascades overwhelm vector database connection pools when multiple agents time out simultaneously, causing fleet-wide retrieval outages where all subsequent agent queries fail until the pool recovers. Partial failure in parallel gather operations (e.g., asyncio.gather without exception isolation) causes agents to proceed on incomplete result sets without alerting, making tool selection decisions based on attacker-influenced partial contexts. Cache invalidation temporal inconsistency across pipeline stages (Stage 1 expiring before Stage 3) causes agents receiving Stage 2-fresh but Stage 1-stale results to mix knowledge states within a single query response. Single-agent parallel retrieval with exclusive database access eliminates cross-agent race conditions; multi-agent shared indexes create irreducible cross-agent concurrency hazards. [NoFW]
- 5.4.9 Multi-Stage Pipeline Result Caching Race Enabling Multi-Agent Consistency Failures: Shared caching infrastructure (Redis, Memcached) serving a multi-agent fleet creates race conditions and stale data when multiple agents simultaneously update or invalidate cached entries. TTL race conditions cause agents with staggered cache expiries to serve contradictory answers to identical queries for 5–30 minute windows depending on routing, enabling targeted inconsistency attacks. Event-based invalidation propagation delays (sequential propagation over minutes) allow agents receiving late invalidation signals to serve stale results indefinitely if invalidation delivery fails. Hash collision attacks exploit normalization-equivalent crafted queries (e.g., inserting zero-width space characters) to poison shared cache entries affecting all agents querying that cache key simultaneously. Cache warming inconsistency where only a subset of agents populate the shared cache on startup creates exploitable cold-agent routing that forces expensive re-queries with attacker-controlled timing. Single-agent exclusive local caches limit race conditions to that agent's concurrent request handling; multi-agent shared caches transform single-entry poisoning into fleet-wide simultaneous impact. [NoFW]
- 5.4.10 Shared Workflow State Version Conflict Amplification Through Concurrent Write Flooding Creating Retry Storm Cascades: Production multi-agent workflows implementing optimistic locking for shared state coordination (version-numbered writes that fail on conflict) exhibit quadratic scaling of conflict rates under concurrent load (e.g., 2% at 10 concurrent workflows scaling to ~50% at 100 concurrent workflows). Targeted attacks submitting bursts of concurrent requests force all agents to the same workflow phase simultaneously, overwhelming the version-conflict resolution system. Automatic retry logic implementing exponential backoff creates retry storms where agents repeatedly re-read, recompute, and re-attempt writes generating positive feedback loops that exhaust retry budgets without progress, causing ~40% of workflows to fail entirely under coordinated attack. The retry-storm cascade is uniquely multi-agent: single-agent workflows perform all state updates sequentially with no cross-agent version conflicts; multi-agent shared state coordination creates irreducible concurrency conflict hazards that attackers can amplify to fleet-wide workflow failure through coordinated request bursts. [NoFW]
- 5.4.11 Guardrail Bypass Through Infrastructure Failure Injection Preventing Safety Validation Execution: Production agentic architectures interposing NeMo Guardrails validation between agent output generation and response delivery configure timeout thresholds preventing indefinite blocking (e.g., 500ms validation timeout). Attackers inject processing delays through crafted inputs requiring maximum validation computation time (dense PII-like patterns, maximum-length text, complex compliance conditions approaching the 880ms validation ceiling on a 350ms baseline). Sustained delay injection causes validation to exceed the configured timeout, triggering automatic bypass where agents deliver unvalidated responses rather than blocking. Service unavailability attacks targeting the validation microservice via network flooding or resource exhaustion induce fleet-wide bypass: all agents configured to fail-open on validation service unavailability deliver unvalidated outputs simultaneously. Unlike saturation-based bypass (which requires sustained load above capacity), failure-injection bypass requires only a brief targeted attack on infrastructure availability or processing time. Single-agent dedicated guardrail instances limit bypass impact to one agent's users; centralized validation services create correlated fleet-wide bypass under attack. [NoFW]
- 5.4.12 Centralized Bias Detection Infrastructure Saturation Creating Fleet-Wide Fairness Monitoring Gaps: Production fairness operations routing all agent outputs through a centralized BiasScorer inference cluster for pre-delivery evaluation face catastrophic fairness monitoring collapse when service saturation exceeds configured agent response timeouts. Partial infrastructure failure (e.g., one of three cluster instances crashing) reduces capacity 33%, shifting queue dynamics so that bias detection latency approaches or exceeds per-agent response timeouts, causing agents to deliver responses without bias evaluation. Adversaries can deliberately trigger cluster saturation by submitting high-complexity requests to all agents simultaneously during periods when legitimate load already approaches capacity. Under saturation, discriminatory outputs reach users without detection, and the bypass is invisible to standard logging that records only agent response delivery (not validation completion). Single-agent deployments with dedicated bias detection avoid queue contention; centralized fleet-wide bias detection infrastructure creates a single saturation point whose failure disables fairness monitoring for every agent simultaneously. [NoFW]
- 5.4.13 Distributed Fairness Metric Tracking Preventing System-Level Bias Detection: Multi-agent deployments monitoring fairness per-agent rather than fleet-wide enable adversaries to distribute discriminatory decisions across agents such that each agent remains individually within per-agent fairness thresholds (e.g., 0–3% demographic parity gap) while the fleet achieves significant aggregate discrimination (e.g., 22% aggregate parity gap for protected subgroups receiving systematically biased routing). Per-agent monitoring lacks statistical power to detect intersectional bias affecting small subgroups within each agent's decision volume (e.g., 25 Black female applicants per agent versus 200 fleet-wide). Adversaries exploiting this fragmentation strategically route protected-group applicants to agents handling majority-group pools, achieving aggregate bias that disappears entirely when disaggregated to per-agent metrics. Single-agent deployments with centralized decision databases enable intersectional analysis across full sample sizes with sufficient statistical power; distributed per-agent tracking creates irremediable blind spots for fleet-level discriminatory patterns that exist only in aggregate. [NoFW]
- 5.4.14 Model Registry Poisoning via Container Image Tampering: MLflow model registries storing versioned agent artifacts can be poisoned by injecting malicious model files; deployments pulling "latest" versions automatically pull the poisoned model. Multi-agent systems with N agents pulling from a shared registry enable a single registry compromise to infect all N agents simultaneously, whereas single-agent deployments with isolated model storage limit blast radius to one deployment. [NoFW]
- 5.4.15 Over-the-Air Update Rollback Manipulation Enabling Tool Injection: Agent deployment systems supporting over-the-air model updates with rollback capabilities enable supply chain attacks through rollback manipulation. Attackers with access to deployment management infrastructure trigger rollbacks to previous versions containing known vulnerabilities or less sophisticated safety training, reverting deployed agents to more exploitable model versions. In multi-agent orchestrated deployments, coordinated rollback of multiple agents to vulnerable previous versions creates a fleet-wide vulnerability window. Unlike forward deployment attacks, rollback attacks exploit the trusted rollback mechanism as a regression vector. [NoFW]
